# 目录结构
## /src/hotspot
JVM的hotspot实现

+ cpu: 特定cpu架构的实现
+ os：特定操作系统的实现
+ os_cpu：特定cpu+操作系统的实现

share: 平台无关的实现，大部分都在该目录

+ include: 公共的头文件，定义了jvm核心的宏、常量、结构
+ libadt: 公共数据结构，比如映射表Dict、集合VectorSet
+ JIT编译器相关的：ci(c1/c2编译器接口)，c1(c1编译器，轻量JIT编译器)，compile（c1和c2的公共逻辑）, opto(c2编译器), jvmci
+ classfile: 字节码文件的加载和解析、验证
+ oops: 对象模型核心实现，比如class类的jvm实现instanceClass
+ runtime: 核心运行时，包括线程管理、内存管理、类加载
+ memory: 内存管理、gc的平台无关部分的代码
+ prims: jni调用

## /src/java.base
+ 是java9的模块，java最核心的模块，所有其他模块都间接依赖它
+ 分为操作系统特定实现和share，平台无关实现

share：分为classes的java实现，和native的c++实现

classes: java标准库API

+ java/lang, java/io, java/math, java/util, java/net, java/nio: 对应jdk9之前的同名包，也是java.base模块的对外导出的包(export java.lang...)，外部应用可以使用，确保向后兼容
+ jdk/internal/*: 对应jdk9之前的sun.*包，是jdk内部使用的包，模块系统禁止外部应用<font style="color:#DF2A3F;">直接</font>访问(exports jdk.internal.misc to java.desktop,...只能to后面的包访问)，不保证向后兼容，比如jdk.internal.misc.Unsafe用于支持java.util.concurrent包，jdk.internal.reflect.Reflection用于支持java.lang.reflect包。可以通过--add-opens java.base/*=ALL-UNNAMED强行访问
+ sun/*: 旧的(jdk1~8)sun/*的很多代码都移动到了jdk/internal中，9之后的sun/*是历史遗留的少量代码

native: java/base中的native实现和启动器的实现，classes依赖native，native依赖hotspot，hotspot依赖操作系统

+ include: 头文件，定义了核心的宏、结构、常量
+ launcher: java命令和tools命令的实现，包含main.c
+ libjli: launcher直接依赖它，用于加载libjvm，
+ libjava, libzip, libnet, libnio: classes的native实现，比如Object.getClass, Thread.sleep, java.util.zip的native等，只包含入口代码，具体实现还是在hotspot

# 启动流程
## java.c
+ JLI_Launch：main.c直接调用的函数，完成所有工作
+ JavaMain ：java主线程对应的函数，会初始化JVM，加载java主类，调用java main函数
+ InitializeJVM：调用libjvm.so中的JNI_CreateJavaVM，创建jvm，并返回JNIEnv，后续可以通过JNIEnv调用java方法，比如NewObjectArray，创建java数组
+ LoadMainClass：通过sun.launcher.LauncherHelper.checkAndLoadMain加载java main class

## java_md.c
+ LoadJavaVM：加载linux动态链接库libjvm.so，读取其中的3个函数，保存到InvocationFunctions，libjvm.so是hotspot的核心库，由src/hotspot中的核心代码编译得到，包含jit，gc, runtime, memory, classfile
+ JVMInit，ContinueInNewThread0：初始化jvm，使用linux线程库pthread创建线程，并join等待

```c
ifn->CreateJavaVM = (CreateJavaVM_t)
```

## jni.cpp
+ JNI_CreateJavaVM: 创建jvm，，具体创建调用Threads::create_vm创建。它只是一个转发的接口，
+ 为什么要将创建jvm的接口放到jni: jni是jvm与本地代码交互的接口，包括创建、获取、销毁jvm，以及java程序通过jvm调用本地代码，所以放到jni

## threads.cpp
+ create_vm: 包括创建线程、分配内存、初始化系统类、初始化系统属性、创建类加载器、初始化静态符号表
+ 为什么具体创建jvm的逻辑放到threads.cpp: 它属于runtime目录，而创建jvm流程包括创建线程、分配内存、初始化系统类，这些都属于运行时
+ initialize_class：确保指定的类已经加载、解析、初始化
+ main_thread：当前线程就是主线程，创建一个main_thread与当前线程绑定(将main_thread保存到当前线程的TLS)
+ init_globals2：初始化解释器(初始化方法入口表)
+ create_initial_thread：创建一个java线程，将c++线程main_thread与java线程双向绑定，java.lang.Thread的eetop指向main_thread，调用java线程的init方法，线程名称是main
+ call_initPhase1: 调用System.initPhase1，初始化系统属性(System.getProperty返回的属性), System.in, System.out
+ call_initPhase2: 调用System.initPhase2，初始化模块化系统，触发ClassLoaders类加载，从而触发类加载器(platformClassLoader, appClassLoader)的创建
+ call_initPhase3: 调用System.initPhase3，根据系统属性java.system.class.loader，创建系统类加载器(SystemClassLoader),并且将它设置为当前线程的上下文类加载器(ContextClassLoader)，如果没有指定系统属性，则使用appClassLoader

## threadLocalStorage_posix.cpp
+ 线程本地存储（TLS），通过pthread_key_create、pthread_getspecific、pthread_setspecific实现
+ pthread_key_create(*thread_key)：线程本地存储是key-value格式的，分配一个threadLocal，key是thread_key
+ pthread_setspecific(thread_key, pointer)：设置key的value，是pointer
+ pthread_getspecific(thread_key): 根据key获取value

## thread.cpp
+ Thread是jvm中的线程，子类JavaThread、VMThread
+ initialize_thread_current：1、将当前Thread对象保存到当前os线程的TLS的thread_key。2、将当前Thread对象保存到静态TLS变量_thr_current（通过static __thread Thread* _thr_current定义，__thread是gcc编译器扩展，用于支持TLS）。这2步都是为了Thread::current()能获取从TLS中获取到这个Thread对象。
+ _handle_area: 是多个内存块，真正的TLS内存空间，用来分配Handle的存储空间，而Handle用来保存java对象的指针

# 通用结构
## handles.hpp, handles.inline.hpp, handles.cpp
+ HandleArea: 是Arena的子类，每个线程对应一个，用来存储Handle的，包含多个Handle，而Handle是用来存储java对象的指针oop的地址
+ Arena：多个内存块，_first是第一个内存块，chunk是当前真正使用的内存块，hwm是高水位线，当前内存块已使用和未使用的边界指针，max当前内存块最大位置的指针

Handle

+ 用于c++中访问java中的对象时，必须通过Handle来访问，直接使用oop会导致gc移动对象而原来的指针错误指向
+ 只有一个成员变量*oop，oop是java对象的指针，*oop是指向指针的指针，gc时，oop的值可能会由于对象移动而改变，但oop本身的地址不变，当gc需要移动对象时，会遍历所有线程的HandleArea中的Handle，修改其中对象地址发送变化的
+ 有多个子类，通过宏定义，比如instanceHandle，通过DEF_HANDLE_CONSTR宏定义

HandleMark: 只用来栈上分配的对象，从而离开当前作用域时，自动执行析构函数。它的构造函数保存线程的HandleArea的内存块的当前分配状态，析构函数恢复线程的HandleArea的内存块的分配状态，从而实现离开作用域自动清空作用域内分配的Handle

```c
HandleMark hm(THREAD); // 将线程的HandleArea的内存块的当前分配状态，保存到栈
Handle h1(THREAD, obj1); // 在HandleArea中分配新的内存空间给Handle
Handle h2(THREAD, obj2);
// 离开作用域，恢复线程的HandleArea的内存块的分配状态，从而清空h1和h2
```

## vmSymbols，symbolTable，symbol
+ 符号表：jvm中常用的字符串，为了防止相同字符串重复存储，jvm启动时就将这些常用字符串保存为符号引用Symbol，将符号引用保存到符号表，使用时直接从符号表取。符号表分为静态符号表和动态符号表
+ 静态符号表：比如系统类名java.lang.Object, Thread,String...，方法签名, B(Byte),C(Char),Z(bool)，Ljava/lang/Runnable;(引用类型)，[Ljava/lang/String;(数组类型)，保存在Symbol::_vm_symbols和symbolTable中
+ 动态符号表：jvm启动后，按需添加进symbolTable的

vmSymbols

+ initialize: 初始化静态符号表，创建jvm时调用，遍历枚举类型vmSymbolID和vm_symbol_bodies，向symbolTable创建并添加Symbol符号引用，并且将符号引用保存到Symbol::_vm_symbols
+ vmSymbolID：静态符号表的枚举，用作符号表数组的索引，通过宏VM_SYMBOLS_DO来创建
+ vm_symbol_bodies：静态符号表要存储的字符串，通过VM_SYMBOLS_DO创建，是一个\0分割的长字符串，每个对应一个静态符号
+ VM_SYMBOL_DECLARE：这个宏定义了静态符号引用的getter方法，比如vmSymbols::java_lang_String()，实际上是根据方法名对应的vmSymbolID枚举从Symbol::_vm_symbols中取出符号引用

symbol

+ Symbol: 符号引用
+ _vm_symbols: 静态符号表，数组索引是vmSymbolID枚举值，值是Symbol符号引用
+ _body: 符号字符串的指针，length是字符串长度

symbolTable

+ SymbolTable: 符号表
+ new_permanent_symbol(name): 根据name字符串，创建并获取对应的静态符号引用
+ lookup_only(name): 根据name查找符号引用
+ do_add_if_needed(name)：根据name创建并添加符号引用

## 常量池
+ class文件的常量池：保存了对应的类用到的字面量，比如：类名、方法名、方法签名、字符串、基本类型值，以常量池项的格式存储，用于复用字面量，减小class文件体积，每个常量池项对应一个常量池项索引
+ 常量池项：包括：项的类型、指向其他常量池项的索引，或者长度+变长字节数组
+ 运行时常量池：类加载的解析阶段，会将class文件中的常量池解析、加载到Klass对象的ConstantPool对象中。它只保存了常量对象（比如Klass、字符串）的指针，先是符号引用，第一次使用，创建对象后，修改为对象的直接引用。作用是加快常量对象的访问速度，避免从全局表中取
+ 符号字面量（Utf8）：类名、方法名、字符串等常量池项最终指向的项，解析时，会将它加到符号表，避免重复，运行时常量池只保存了符号对象的指针
+ 字符串字面量(String)：类中用到的字符串常量，解析时，保存的是它的符号对象的指针，ldc加载字符串常量时，才会真正创建字符串对象，并加到字符串表
+ 符号表通过引用计数法来判断存活，常量池添加或获取符号后，符号的引用计数+1，类卸载删除常量池时，-1，为0时删除符号，符号在c堆中分配

## 字符串
+ 字符串常量的对象通过ldc创建，它先获取当前栈帧的方法的类的常量池，调用常量池加载字符串对象
+ ldc字符串时，会先尝试从常量池缓存中取已经加载的对象引用，如果没有，才会获取符号引用，将符号引用intern到字符串表，然后将字符串表中的字符串对象保存到常量池缓存，返回对象
+ jdk9开始，字符串内部使用字节数组，而不是字符数组，对于ISO-8859-1编码的字符串，使用1字节存储字符，其他情况仍utf16
+ 表的节点通过Weakhandle持有字符串对象，通过Weakhandle.oop是否等于null来判断是否被gc回收，如果回收，该节点死亡，随后清除
+ 字符串DeDup: 配置了-XX:+UseStringDeduplication时，启动阶段会启动1个StringDedup线程，在young/mixed/full gc的标记阶段，会将分代年龄小于阈值（StringDeduplicationAgeThreshold，默认3）的java String对象加到待Dedup列表，StringDedup线程扫描它，通过一张key是字节数组的hash表，将值相同的String对象的内部字节数组替换成同一个
+ 为什么只对分代年龄小的String进行Dedup：分代年龄大的String很可能已经在Dedup表中，防止每次gc都对所有String做Dedup

## rcu
+ 是一种读写锁，通过版本号控制读写并发。读无锁，进入读临界区时，将全局版本号保存到当前读线程局部版本号，退出临界区时，恢复局部版本号。写分为2个过程，
1. 第1个过程，创建一个新的版本（比如写时复制），原子切换到新版本，之后进入读临界区的线程只能读到新版本；
2. 第2个过程，写线程将全局版本号+1，从次进入读临界区的线程获取到的是新版本号，读到的是新版本，写线程等待持有老版本号的读线程退出临界区，这时，可以安全的删除老版本号对应的数据
+ 读线程无锁，写线程可以新版本的读线程并发执行，适合读多写少

```cpp
void InterpreterMacroAssembler::dispatch_next(TosState state, int step, bool generate_poll) {
  // load next bytecode
// rbcp(bytecode pointer)寄存器保存了当前字节码的地址，pre(rbcp, step)相当于rbcp=rbcp+step，取下一个字节码地址
    //ldrb（load register byte）是加载一个字节到寄存器
    //相当于rbcp=rbcp+step, rscratch1 = *(uint8_t*)rbcp, rscratch1最终就是当前执行的字节码的opcode值(0~255)
  ldrb(rscratch1, Address(pre(rbcp, step)));
  dispatch_base(state, Interpreter::dispatch_table(state), /*verifyoop*/true, generate_poll);
}
void InterpreterMacroAssembler::dispatch_base(TosState state,
                                              address* table,
                                              bool verifyoop,
                                              bool generate_poll) {
    // rscratch2 = rscratch1 + Interpreter::distance_from_dispatch_table(state)
    // 分派表是一个2维数组，但连续存储，第一个维度是状态，第2个维度是0~255的字节码，根据字节码值和状态得到分派表的索引
    addw(rscratch2, rscratch1, Interpreter::distance_from_dispatch_table(state));
    //rscratch2 = *(rdispatch + (rscratch2 << 3))
    // 根据分派表起始地址rdispatch和偏移量(索引rscratch2)，得到分派表中的值，也就是字节码对应的模版解析器的入口地址
    ldr(rscratch2, Address(rdispatch, rscratch2, Address::uxtw(3)));
    // 跳转执行字节码，这里面最后一般会调用dispatch_next，取下一个字节码，查分派表，循环执行(解释器循环)
    br(rscratch2);
}
```

+ 寄存器优化: 多个入口是为了寄存器优化，比如iconst指令实际上不会直接将操作数压栈，因为写入内存很耗时，而是mov指令将操作数保存到r0，r0是用来保存当前栈顶操作数的值的寄存器，同时将TOS置为itos，第2条iconst指令执行时，因为TOS=itos，需要先将当前栈顶值寄存器，也就是r0，压入栈顶，再将2写入r0。2条都是iconst指令，但入口地址不同
+ 实际栈顶操作数：栈顶+寄存器r0, r0是栈顶第1个，内存中的栈顶是栈顶第2个，压栈指令需要根据TOS判断栈顶(r0)是否有值，如果有，需要先保存上一个指令压入的栈顶操作数
+ str(store reg)，从寄存器写入内存, ldrw(load reg word)，从内存写入寄存器，写1个32位的word

```cpp

iconst_2   // 执行前：TOS=itos，r0=1，走iep入口，先str(r0, sp),再mov(r0, 2), 执行后：TOS=itos, r0=2, *sp=1
iadd       // ldrw(r1, sp),andw(r0, r1, r0), 执行后：TOS=itos, r0=3
```

# G1
+ g1是一个面向多处理器、较大内存的垃圾收集器，以停顿时间为目标，实现低延迟、高吞吐的垃圾收集，能够较好的平衡延迟和吞吐量
+ 是jdk9~25的默认收集器，jdk8默认是parallel
+ g1是通过在gc后，调整年轻代大小来控制STW的，因此不建议使用-Xmn将年轻代大小固定，这样会时自适应调整年轻代大小的策略失效，从而无法控制STW，一般只配置-Xms和-Xmx，年轻代大小使用默认值(Xmx的5%~60%)，或者-XX:NewSize和-XX:NewSizeMax使用不同的值
+ MaxGCPauseMillis：1、单次gc暂停时间不能超过的值。2、在任意时间窗口(GCPauseIntervalMillis，默认201ms)内，允许的暂停总时间不超过的值(MaxGCPauseMillis，默认200ms)。MaxGCPauseMillis值调大会使gc次数变少，吞吐量上升，但延迟可能增加；调小则相反。
+ GCPauseIntervalMillis：调大，相当于MaxGCPauseMillis/GCPauseIntervalMillis减小，也就是单位时间gc的时间减小，即gc频率减小，每次gc的时间增加，eden增大；调小相反
+ 软引用清理策略：LRU策略，更少get访问的更容易被清理，而不是堆内存紧张时才清理。java SoftReference对象内维护一个时间戳t，每次get()访问时更新，虚拟机维护一个全局的时间戳t1，每次young/mixed/full gc时更新，t1-t>max_interval时，清理该软引用，max_interval=max_heap(单位m)*SoftRefLRUPolicyMSPerMB(默认1000ms)。因此软引用不适合做缓存
+ young/mixed gc时，复制对象的同时，标记对象，所以可以直接在mark word中标记，标记是为了防止重复复制；并发标记时，不能占用mark word，因此使用标记位图；full gc时，标记和规整是分开进行的，规整时，需要遍历region中的存活对象，使用标记位图可以提高遍历的效率

+ 预留空间：通过mmap+PROT_NONE实现，内部只在vma申请一块虚拟地址范围，不会创建页表项，预留空间读写会报内存页错误sigsegv
+ 为什么使用预留空间，而不按需commit/uncommit：g1收集器强依赖一块连续的堆内存，如果按需commit/uncommit，会导致堆空间不连续、收集器实现过于复杂
+ commit使用mmap+PROT_READ|PROT_WRITE，commit后，os会创建页表项，但不会立即申请物理页，而是等到读写对应空间时，触发缺页中断，申请物理页，可以使用-XX:+AlwaysPreTouch，commit后向每个页读写一次，从而在启动阶段提前申请物理页，避免运行时申请，导致创建对象卡顿、gc停顿过长
+ uncommit使用mmap+MAP_NORESERVE, uncommit后，物理页释放、页表项清空，回到预留状态，读写会报内存页错误

```cpp
//jvm 预留空间底层实现
static char* anon_mmap(char* requested_addr, size_t bytes) {
  // MAP_FIXED is intentionally left out, to leave existing mappings intact.
  const int flags = MAP_PRIVATE | MAP_NORESERVE | MAP_ANONYMOUS;

  char* addr = (char*)::mmap(requested_addr, bytes, PROT_NONE, flags, -1, 0);

  return addr == MAP_FAILED ? nullptr : addr;
}
int os::Linux::commit_memory_impl(char* addr, size_t size, bool exec) {
  int prot = exec ? PROT_READ|PROT_WRITE|PROT_EXEC : PROT_READ|PROT_WRITE;
  uintptr_t res = (uintptr_t) ::mmap(addr, size, prot,
                                     MAP_PRIVATE|MAP_FIXED|MAP_ANONYMOUS, -1, 0);
}
bool os::pd_uncommit_memory(char* addr, size_t size, bool exec) {
  uintptr_t res = (uintptr_t) ::mmap(addr, size, PROT_NONE,
                                     MAP_PRIVATE|MAP_FIXED|MAP_NORESERVE|MAP_ANONYMOUS, -1, 0);
  return res  != (uintptr_t) MAP_FAILED;
}
```

```cpp
// 根据当前时间和预期下一次暂停时间，得到下一次stw距离当前时间的间隔时间
double G1MMUTracker::when_sec(double current_timestamp, double pause_time) {
  // If the pause is over the maximum, just assume that it's the maximum.
  pause_time = MIN2(pause_time, max_gc_time());
//历史时间窗口内，允许的总暂停时间
  double gc_budget = max_gc_time() - pause_time;
//时间窗口的开始时间
  double limit = current_timestamp + pause_time - _time_slice;
  // Iterate from newest to oldest.
  for (int i = 0; i < _no_entries; ++i) {
      //_array是一个环形数组，保存了最近一段时间，每次stw的开始和结束时间，从最近的往最远的遍历
    int index = trim_index(_head_index - i);
    G1MMUTrackerElem *elem = &_array[index];
    // Outside the window.超出时间窗口了，
    if (elem->end_time() <= limit) {
      break;
    }
//单次stw的时间
    double duration = (elem->end_time() - MAX2(elem->start_time(), limit));
    // This duration would exceed (strictly greater than) the budget.
      //预算时间不够了，balance_timestamp开始的事件窗口恰好能满足预算时间，balance_timestamp - limit，就是还能有多久不能暂停，也就是java线程能够运行多久时间
    if (duration > gc_budget) {
      // This timestamp captures the instant the budget is balanced (or used up).
      double balance_timestamp = elem->end_time() - gc_budget;
      assert(balance_timestamp >= limit, "inv");
      return balance_timestamp - limit;
    }

    gc_budget -= duration;
  }

  // Not enough gc time spent inside the window, we have a budget surplus.
  return 0;
}
// 预测的eden区分配速率*java线程能运行的时间，就是mmu所需的eden区的大小(延迟优先)
uint G1Policy::calculate_desired_eden_length_by_mmu() const {
  assert(use_adaptive_young_list_length(), "precondition");
  double now_sec = os::elapsedTime();
  double when_ms = _mmu_tracker->when_max_gc_sec(now_sec) * 1000.0;
  double alloc_rate_ms = _analytics->predict_alloc_rate_ms();
  return (uint) ceil(alloc_rate_ms * when_ms);
}
// 在eden区配置值的范围内，取一个尽可能大的、能满足最大暂停时间的eden大小（吞吐量优先）
uint G1Policy::calculate_desired_eden_length_before_young_only(double base_time_ms,
                                                               uint min_eden_length,
                                                               uint max_eden_length) const {
    //base_time_ms包括了卡表合并、扫描、survivor复制时间
  G1YoungLengthPredictor p(base_time_ms,
                           _free_regions_at_end_of_collection,
                           _mmu_tracker->max_gc_time() * 1000.0,
                           this);
    //根据eden大小得到eden复制的时间，加上基础时间，是否满足暂停目标
  if (p.will_fit(min_eden_length)) {
    if (p.will_fit(max_eden_length)) {
      min_eden_length = max_eden_length;
    } else {
      //2分法查找一个满足条件的值
      uint diff = (max_eden_length - min_eden_length) / 2;
      while (diff > 0) {
        uint eden_length = min_eden_length + diff;
        if (p.will_fit(eden_length)) {
          min_eden_length = eden_length;
        } else {
          max_eden_length = eden_length;
        }
        assert(min_eden_length <  max_eden_length, "invariant");
        diff = (max_eden_length - min_eden_length) / 2;
      }
    }
  }
  return min_eden_length;
}
```

## TLAB
+ 线程分配对象时，会额外多申请一段空间，多申请的空间就是tlab，tlab不能用于其他线程分配对象，后续分配对象时，线程可以直接使用tlab，而不需要cas锁或堆锁
+ 如果没有指定TLABSize，会把整个eden区都当做tlab，每个正在分配对象的线程平均分eden区，但不会一次性分给线程，而是分多次(_target_refills,默认50)，线程每次申请分配新的tlab的过程称为refill
+ tlab也是在region上分配，不能超过humongons大小，也就是region的一半，线程只维护最近一次分配的tlab，refill时，老的tlab退役(不是回收)，退役的tlab的空闲空间会造成浪费(waste)，线程结束并且gc前都无法使用，另外refill次数过少，浪费的多，refill次数过多，慢路径多
+ Thread.start创建os线程后，执行线程函数，其中会初始化tlab
+ tlab初始化：包括启动时tlab全局变量的初始化，和java线程启动时，对应线程的tlab的初始化

```cpp
Universe::initialize_tlab // tlab全局变量的初始化
    ThreadLocalAllocBuffer::startup_initialization //初始化全局变量_allocating_threads_avg，用于统计正在分配内存的线程数的算数均值，_target_refills，每个线程允许refill次数

thread_native_entry //线程tlab的初始化
    Thread::call_run
        JavaThread::run // 线程入口函数会调用JavaThread的回调函数
            initialize_tlab // run之前，初始化当前线程的tlab
            thread_main_inner// 调用JavaThread::entry_point,从而调用java线程的run方法
```

+ java线程创建时，只创建了java.lang.Thread对象，Thread.start才会在jvm中创建名为JavaThread的c++对象，并且pthread_create创建os线程

```cpp

    JavaThread::JavaThread(ThreadFunction entry_point, size_t stack_sz)// 保存entry_point，这里的entry_point会调用java线程的run方法
        os::create_thread// 调用pthread_create创建os线程，将线程入口函数设为thread_native_entry
```

+ tlab的空间分配：在线程第一次创建对象时，才会为tlab分配空间，先尝试在已有的tlab上分配，再尝试分配新的tlab，在新的tlab上分配

```cpp
        mem_allocate_outside_tlab // 不在tlab上面分配
```

## safepoint
+ 安全点(STW)是通过VMThread发起的，当它接收到的VM_Operation需要安全点时(VM_Operation.evaluate_at_safepoint返回true)，就会通过SafepointSynchronize.begin发起安全点请求，修改polling_word为_poll_word_armed_value，其他线程在执行到特定的地方会检查polling_word，如果需要STW，则调用process_if_requested，它使用futex实现线程等待
+ 线程会检查安全点的地方：
1. goto，tableswitch, lookupswitch字节码指令，对应循环每一轮的结束位置、switch，它们会检查polling_word，如果需要stw，则先不使用分派表，而是使用_safept_table，跳转到InterpreterRuntime::at_safepoint，它是一个JRT_ENTRY，该宏在退出时，通过析构函数调用SafepointMechanism来检查和进入安全点
2. 从一些vm返回到java方法时，也是通过JRT_ENTRY

## 并发标记
+ 负责老年代对象的标记，之后会进入mixed gc，真正回收老年代对象
+ IHOP，Initiating Heap Occupancy Percent，Initiating表示并发标记开始，老年代的占用比例达到这个门限时，开启并发标记。自适应IHOP会根据运行时间动态调整
+ 为什么需要IHOP：并发标记不会stw，在标记的同时java线程也在分配新的空间，如果开启太晚，并发标记未运行结束时，java线程就空间不足了，会导致full gc；如果开启太早，导致回收效率低
+ 如何计算自适应IHOP：根据mutator的分配速率*并发标记的运行时间，得到并发标记期间mutator需要的空间大小的预测值p，max_heap-p，得到并发标记最迟应该开始的堆占用空间大小，除以max_heap
+ 并发标记开始前，会经过2轮young gc，第一轮结束时，负责判断IHOP门限，第2轮开始时，开启conc_start状态，第2轮结束时，关闭conc_start状态，通知并发标记线程
+ SATBMarkQueue：Snapshot-At-The-Beginning，并发标记只标记并发标记开始时那一刻的对象图。因为在遍历对象图时，java线程也在运行，可能会删除或新增对象字段引用，对于删除的引用，需要使用SATB队列记录它们；对于新增的引用，不处理，默认为存活。

## 元空间
+ class mirror: java Class对象，元空间的Klass对应的镜像
+ 元空间分割成多个块（Chunk）来管理，类似于堆的region，但大小不固定，分为15个层级，1K~16M，2的指数倍增长。申请空间时，先获取最大的16M块（root块），再不断地对半分割，直到刚好满足分配请求，将其他块放回空闲块列表，用于后续申请时直接获取。释放空间时，会先合并空闲的块，再放回空闲块列表
+ leader块：分割块时，低地址的块是leader，高地址是follower，只有leader可以通过合并follower来扩容，因为1、使用了碰撞指针，空间分配只能向单个方向增长。2、简化设计。2个块互为伙伴(buddy)
+ 活跃块：类似于堆的活跃region，是当前正在使用的块，通过碰撞指针来分配，活动块空闲空间不足时，如果活跃块是leader，先尝试扩容。元空间是分配时，按需提交内存，<font style="color:#DF2A3F;">MetadataSize</font>是触发gc的门限，而不是初始化提交的大小。<font style="color:#DF2A3F;">MaxMetadataSize</font>是元空间oom的门限
+ 元空间分为类元空间和非类元空间，类元空间用来保存Klass对象，非类元空间用来保存其他对象。因为如果使用压缩指针，类元空间需要一块连续的内存区域，所以类元空间会先预留一块固定大小、不可扩展的内存区域(CompressedClassSpaceSize，默认1G)，而非类元空间大小可以扩展，每次扩展16M。如果类元空间达到<font style="color:#DF2A3F;">CompressedClassSpaceSize</font>，会报OutOfMemoryError: Compressed class space，其他情况的元空间oom报OutOfMemoryError: Metaspace
+ 元空间的<font style="color:#DF2A3F;">连续空间</font>达到MetadataSize（默认21M），并且提高该门限值时，门限值达到MaxMetadataSize(默认max_int)，触发full gc
+ 如果开启ClassUnloadingWithConcurrentMark（默认开启），并发收集阶段会进行类卸载：
1. 并发标记前的young gc不会对非bootstrap类加载器中的老年代的java Class和ClassLoader做保活标记
2. 并发标记遍历对象图时，获取对象的Klass指针的类加载器，对CLD下的所有java Class, Classloader进行标记保活
3. 重标记完成后，遍历所有WeakHandle，判断CLD的java Classloader是否标记，如果未标记，将WeakHandle的oop置空。将所有WeakHandle为空的CLD加到待卸载链表，然后是否对应的已分配空间
+ 类卸载的条件：通过类加载器是否被标记为垃圾对象来判断的，而java Class对象通过强引用持有了类加载器，java对象会对所属类的类加载器的所有java Class对象、类加载器对象进行保活标记

## full gc
+ 触发条件：
1. 达到eden大小的门限，触发young/mixed gc，eden region复制失败(已提交的空闲空间不满足复制对象所需)，导致垃圾对象无法回收，然后堆扩容至Xmx后，仍无法满足分配请求
2. 达到eden大小的门限，触发young/mixed gc，eden中大部分都是存活对象，导致回收率低，然后堆扩容至Xmx后，仍无法满足分配请求
3. 元空间的<font style="color:#DF2A3F;">连续空间</font>达到MetadataSize，并且提高该门限值时，门限值达到MaxMetadataSize，触发full gc
4. System.gc()，并且未开启DisableExplicitGC和ExplicitGCInvokesConcurrent，会至少触发一次full gc
5. jmap -dump，或者jcmd GC.heap_dump中指定了live，会先full gc再dump
+ 规整（compact）：把存活对象从堆的起始地址开始，顺序存放，从而减少空间碎片
+ 预规整：目的是让串行规整能够复用并行的结果，如果直接规整，只能全部并行或全部串行。预规整是在真正复制对象前，确定每个存活对象的目标地址，将它保存到mark word，因此需要先将原来的mark word暂存起来，规整结束后恢复
+ 标记-规整：因为规整需要按照堆地址顺序，比如从低地址向高地址的顺序遍历存活对象，将它们向低地址的方向移动，所以规整前，需要先从gc roots开始，在标记位图中标记存活对象
+ full gc有2个选项
1. 是否清理所有的软引用，包括未到LRU过期时间的
2. 是否最大化规整：n个线程并发规整后，会产生n个碎片，最大化规整会在并发规整的基础上，执行串行规整，最大化减少碎片
+ humongous规整：清理+规整，尽可能减少大对象的复制。标记完成后，直接将垃圾H对象占用的region设为空闲，存活H对象如果在其他存活对象中间，不复制；如果在所有存活对象后面，则需要进行规整
