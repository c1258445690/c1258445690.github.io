## 建立、处理连接流程
1. 启动时，会根据配置的bind和port创建socket、bind和listern，将fd加入epoll，监听读事件，注册读回调(这里是连接事件回调)为ConnectionType.accept_handler
2. 监听到连接事件时，执行ConnectionType.accept_handler，执行accept得到连接的fd，根据fd创建connection，根据connection创建client。将fd加入epoll，监听读事件，注册读回调为ConnectionType.aeHandler
3. 连接监听到读事件时，执行ConnectionType.aeHandler，**readQueryFromClient**读取和处理客户端命令，将命令解析到client.argv，从server.commands中获取redisCommand对象，执行redisCommand.proc

**客户端标志位**：保存在client.flags中

+ CLIENT_TYPE_NORMAL -> Normal client
+ CLIENT_TYPE_SLAVE  -> Slave
+ CLIENT_TYPE_PUBSUB -> Client subscribed to Pub/Sub channels
+ CLIENT_TYPE_MASTER -> The client representing our replication master.

## sds
**object_encoding**: 是redisObject的子类型

+ OBJ_ENCODING_EMBSTR：字符串长度小于等于44，redisObject和sds分配在同一内存块中，只需要一次内存分配和释放，64B是为了匹配缓存行的大小。robj *o = zmalloc(sizeof(robj)+sizeof(struct sdshdr8)+len+1); 16(robj)+3(sdshdr8)+44(string)+1('\0')，注意：sds的几种结构的最后一个buf是伸缩性数组成员，不会为它分配空间，而它指向的空间紧跟在在sds的后面
+ OBJ_ENCODING_RAW：字符串长度大于44，redisObject和sds分开创建，ptr指向sds
+ OBJ_ENCODING_INT：字符串长度小于20，并且能用long类型表示（long最多19位数，加上符号20位），则不使用sds存储，而是转换成long，ptr指向long

### 删除
key的删除有同步和异步2种方法

+ 同步删除：在主线程中释放内存，将要删除的dictEntry节点的前驱的next指针指向要删除的后继
+ 异步删除(unlink)：主线程只将dictEntry.value置为null，这样后面执行valueDestructor时就不会真正释放内存，然后创建释放内存的任务，添加到bio_lazy_free线程的任务队列中，bio_lazy_free线程负责释放内存。

异步删除的条件：value是hash、set、list、zset、stream、module，并且元素个数大于64时，才会执行异步删除

key不会异步删除，只有value会

除了显示指定unlink，还可以通过配置来默认异步删除：


```c
dictEntry *dictTwoPhaseUnlinkFind(dict *d, const void *key, dictEntry ***plink, int *table_index) {
    uint64_t h, idx, table; 
    for (table = 0; table <= 1; table++) {
        idx = h & DICTHT_SIZE_MASK(d->ht_size_exp[table]);
        dictEntry **ref = &d->ht_table[table][idx];
        while (ref && *ref) {
            void *de_key = dictGetKey(*ref);
            if (key == de_key || dictCompareKeys(d, key, de_key)) {
                *table_index = table;
                *plink = ref;//将调用者传入的plink修改为ref
                dictPauseRehashing(d);
                return *ref;
            }
            ref = dictGetNextRef(*ref);//ref是dictEntry.next的指针，用于修改next指向
        }
        if (!dictIsRehashing(d)) return NULL;
    }
    return NULL;
}
```

## set
set有3种实现，通过redisObject.encoding区分：

### OBJ_ENCODING_INTSET
元素个数小于<font style="color:#DF2A3F;">set-max-intset-entries</font>（默认512），并且都是64位范围内的有符号整数时，用intset实现，intset是长度可变、编码类型可变的、整型数组，3种元素编码类型：16位（-32768~32767）、32位、64位，从小到大排序，插入、查找、删除使用二分法查找，增删元素会realloc。<font style="color:#DF2A3F;">使用时尽量添加同种编码类型的整数</font>。

### OBJ_ENCODING_LISTPACK
包含非数值类型，元素个数小于<font style="color:#DF2A3F;">set-max-listpack-entries</font> 128，最大元素占用字节数小于<font style="color:#DF2A3F;">set-max-listpack-value</font> 64。[HDR]+[若干元素...]+[1字节尾部EOF]，HDR=[4字节的ListPack总字节数]+[2字节的元素个数]，[entry]=[加上编码头的元素值]+[backlen，不包括backlen的元素字节数]

单个元素可以存储整型或字符串类型，高位都会加上<font style="color:#DF2A3F;">编码头</font>用于区分不同类型、不同长度范围

+ 0x：7位无符号整数，1B
+ 110xxxxx-x：13位整数，2B
+ 11110001-x-x：16位整数，3B
+ 11110010-x-x-x：24位整数，4B
+ 11110011-x-x-x-x：32位整数，5B
+ 11110100-x-x-x-x-x-x-x-x：64位整数，9B
+ 10xxxxxx-str：小于64B的字符串，x是字节数，默认元素长度小于64B，所以只会用到这个
+ 1110xxxx-x-str：小于4096B的字符串，
+ 11110000-x-x-x-x-str：其他长度的字符串

### OBJ_ENCODING_HT
dictType是setDictType的dict，其中的dictEntry实际上指向的是没有value指针的dictEntryNoValue，当没有链表的下一个元素时，还会直接指向sds类型的key，通过标记指针来区分，因为底层是按字寻址的，64位的机器字长为2B，地址的低2位不会用来寻址，可用于标记。低位为01表示指向sds类型的key，为10表示指向dictEntryNoValue

### 类型转换
+ intset转hashtable：元素个数超过<font style="color:#DF2A3F;">set-max-intset-entries</font>（默认512），或者添加了非数值并且超过Listpack的限制
+ intset转listpack：添加了非数值，并且当前最大元素字节数小于<font style="color:#DF2A3F;">set-max-listpack-value</font> 64、元素个数小于<font style="color:#DF2A3F;">set-max-listpack-entries</font> 128
+ listpack转hashtable：超过了限制

删除元素时，不会反向转换

过期删除策略分为惰性删除和主动删除，惰性删除就是读取key时，判断是否过期

+ 主动删除有2种模式：FAST模式和SLOW模式，它们都在主线程中执行，FAST和SLOW的执行时长的限制不同，前者较小，后者较大，将待过期的key（redisDb.expires）分成多个小部分，每次处理完一小部分后，判断执行时间是否超过时长限制，如果超过则不再执行过期处理；或者上一次回收的key占比是否超过某个值，没超过则不再处理。
+ FAST模式是在epoll_wait之前执行(beforeSleep)，SLOW模式在epoll_wait，处理完所有IO事件之后执行(processTimeEvents)，都只会在master上执行，所有数据库的过期key占比小于10%时，不执行过期；距离上一次FAST模式小于2ms，不过期
+ SLOW模式通过定时任务触发，定时任务在主线程中执行，因为要修改Redis数据结构，只能在单线程中执行，遍历定时任务链表，执行其中触发时间小于当前时间的，其中有一个执行逻辑是ServerCron的定时任务，初始执行间隔是1ms，它调用databasesCron，最后调用<font style="color:#DF2A3F;">activeExpireCycle</font>，传入SLOW模式
+ epoll_wait等待的超时时间不是固定的，等于最近要执行的定时任务的时间戳-当前时间，如果没有定时任务，则无限等待
+ 定时任务serverCron可以通过频率（<font style="color:#DF2A3F;">hz</font>, dynamic-hz）来设置执行间隔，返回值是1000/server.hz，hz越大，该值越小，执行间隔越小，该任务执行越频繁，CPU占用越高，key过期更精确，默认是10，也就是100ms执行间隔，会根据客户端连接数动态变化（dynamic-hz=yes），连接数多时，认为任务量更大，所以增大hz
+ active-expire-effort: effort=active-expire-effort-1，每次只会扫描20+20/4*effort个key，然后判断是否达到时间间隔，时间间隔=(25+2*effort)*1000000/hz/100，FAST时间间隔固定为1ms，上一次回收的过期key占比大于(10-effort)%，才会执行下一次过期，<font style="color:#DF2A3F;">该配置控制了已过期key的占比，调大该值能处理更多已过期的key，减少内存占用，但增大CPU占用</font>。占比是根据上一次回收的已过期的key/扫描的样本数，得到的估计值。因此扫描不是顺序扫描，数组的索引值不是自增1，而是高位自增的方式，比如：0000000000(0)，0000010000(16)，0000001000(8)，0000011000(24)，0000000100(4)。这样做的目的是确保扫描的样本较为均匀覆盖


# 淘汰(eviction)
+ **<font style="color:#DF2A3F;">生产环境一定要配置maxmemory，不能使用默认配置</font>**。redis默认配置是0，不启用淘汰机制(maxmemory-policy=noeviction)，可能会占满整个地址空间
+ 当配置了maxmemory时，会执行performEvictions()，返回EVICT_FAIL时，内存溢出，拒绝flags标志为CMD_DENYOOM的命令，多为写命令
+ 淘汰机制的理想状态是淘汰将来最晚用到的key，这种情况下缓存命中率最高，LRU取空闲时间最大（最长时间未被访问）的key，

| C | | |  | C | | | | C | |


## performEvictions
返回值：

+ EVICT_OK：淘汰成功，内存占用已小于maxmemory，或者不是master节点而是slave，不需要主动淘汰，而是master淘汰后同步给slave；
+ EVICT_FAIL：淘汰失败，拒绝写命令，当内存占用超过maxmemory，并且策略是no-eviction时返回；
+ EVICT_RUNNING：淘汰执行时间超过阈值，开启定时任务(startEvictionTimeProc)执行淘汰，阈值跟maxmemory_eviction_tenacity成正向关系，越大，淘汰能容忍的占用时间越大

要淘汰的内存=申请的堆内存-主从复制和aof缓冲-maxmemory

server.maxmemory_policy是一个位图bitMap，不同淘汰策略用不同的二进制位表示，便于判断"或"条件，将淘汰策略分为3大类：

1. lru或lfu
2. 最小ttl
3. 随机

```c
#define MAXMEMORY_VOLATILE_LRU ((0<<8)|MAXMEMORY_FLAG_LRU)
#define MAXMEMORY_VOLATILE_LFU ((1<<8)|MAXMEMORY_FLAG_LFU)
#define MAXMEMORY_ALLKEYS_LRU ((4<<8)|MAXMEMORY_FLAG_LRU|MAXMEMORY_FLAG_ALLKEYS)
#define MAXMEMORY_ALLKEYS_LFU ((5<<8)|MAXMEMORY_FLAG_LFU|MAXMEMORY_FLAG_ALLKEYS)
if (server.maxmemory_policy & (MAXMEMORY_FLAG_LRU|MAXMEMORY_FLAG_LFU)||
            server.maxmemory_policy == MAXMEMORY_VOLATILE_TTL){...}//策略为volatile_lru,volatile_lfu,allkeys_lru,allkeys_lfu,volatile_ttl
else if(server.maxmemory_policy == MAXMEMORY_ALLKEYS_RANDOM ||
                 server.maxmemory_policy == MAXMEMORY_VOLATILE_RANDOM)//随机策略
```

+ 淘汰分多轮进行，每一轮<font style="color:#DF2A3F;">随机选择固定个数</font>样本，按照空闲时间的顺序放入淘汰内存池，淘汰其中空闲时间最大的key，然后进行下一轮，继续选择固定个数样本放入放入淘汰内存池，淘汰其中空闲时间最大的key，<font style="color:#DF2A3F;">每一轮只淘汰一个key</font>，直到释放内存大小大于等于要释放的内存数
+ 每一轮选择的样本数由<font style="color:#DF2A3F;">maxmemory-samples</font>决定，越大越接近对应淘汰策略的理想情况，默认为5，为10时已经非常接近理想LRU了，但会占用更多CPU
+ 删除key可以惰性删除，lazyfree-lazy-eviction
+ 淘汰执行时间超过阈值，开启定时任务进行淘汰，定时任务立即执行，间隔为0

## evictionPoolPopulate
从所有key或配置了过期时间的key中，随机选取maxmemory-samples个样本，按照淘汰策略的顺序加入淘汰内存池

```c
struct evictionPoolEntry {
    unsigned long long idle;    /*  lru：空闲时间；lfu：255-频率；ttl：Long_MAX-过期时间戳*/
    sds key;                    /* Key name. */
    sds cached;                 /* Cached SDS object for key name. */
    int dbid;                   /* Key DB number. */
};

```

+ ttl：当前系统时间戳-key的访问时间戳，不会每次都调用gettimeofday，而是在定时任务serverCron中定时更新，更新间隔是1000/server.hz

### LFU
频率通过最近的访问次数来表示，每次访问次数+1，并周期性衰减。

+ lfu-log-factor：因为频率次数的最大值只有255，达到最大值不再增加，为了控制增加的速率，每次访问会计算一个概率值，满足时才回+1。增加的速率由<font style="color:#DF2A3F;">lfu-log-factor</font>决定，值越大增加的越慢，0~255覆盖的范围越大，但表示的精度变低，反之亦然，可以先取默认值10，后续根据系统访问速率来决定
+ lfu-decay-time：衰减一次的时间间隔，每几分钟衰减1次，默认1

具体实现：

+ lfu的信息分为2部分，都保存在redisObject.lru中，其中低8位保存了访问频率次数，高24位保存了上一次衰减的分钟时间戳，频率初始值为5

```c
robj *createObject(int type, void *ptr) {
    robj *o = zmalloc(sizeof(*o));
    o->type = type;
    o->encoding = OBJ_ENCODING_RAW;
    o->ptr = ptr;
    o->refcount = 1;
    /* Set the LRU to the current lruclock (minutes resolution), or
     * alternatively the LFU counter. */
    if (server.maxmemory_policy & MAXMEMORY_FLAG_LFU) {
        o->lru = (LFUGetTimeInMinutes()<<8) | LFU_INIT_VAL;
    } else {
        o->lru = LRU_CLOCK();
    }
    return o;
}
void updateLFU(robj *val) {
    //衰减频率值
    unsigned long counter = LFUDecrAndReturn(val);
    //增加频率值
    counter = LFULogIncr(counter);
    //更新衰减时间戳和频率值
    val->lru = (LFUGetTimeInMinutes()<<8) | counter;
}
//evictionPoolPopulate获取频率时也会调用
unsigned long LFUDecrAndReturn(robj *o) {
    unsigned long ldt = o->lru >> 8;
    unsigned long counter = o->lru & 255;
    unsigned long num_periods = server.lfu_decay_time ? LFUTimeElapsed(ldt) / server.lfu_decay_time : 0;
    if (num_periods)
        counter = (num_periods > counter) ? 0 : counter - num_periods;
    return counter;
}
//先获取0~1之间的随机数，再计算1/(频率*lfu_log_factor+1)，如果大于随机数，则增加频率
uint8_t LFULogIncr(uint8_t counter) {
    if (counter == 255) return 255;
    double r = (double)rand()/RAND_MAX;
    double baseval = counter - LFU_INIT_VAL;
    if (baseval < 0) baseval = 0;
    double p = 1.0/(baseval*server.lfu_log_factor+1);
    if (r < p) counter++;
    return counter;
}
//距离上一次衰减的分钟数
unsigned long LFUTimeElapsed(unsigned long ldt) {
    unsigned long now = LFUGetTimeInMinutes();
    if (now >= ldt) return now-ldt;
    return 65535-ldt+now;
}
```

## 淘汰内存池(EvictionPoolLRU)
+ 声明和初始化：是长度为16的evictionPoolEntry数组，其中保存了随机获取的key，空闲时间，所在的dbid，redis启动时，initServer时会初始化它，预先为cached分配好内存，如果key的长度小于255，则可以直接使用cached缓存的空间，避免了反复申请和释放内存(sdsdup和free)，而是memcopy复制内容
+ 插入：从第一个元素开始遍历，找到第一个idel大于插入元素的位置，在这个位置之前插入，如果最后一个元素为空，则从插入位置开始整体向后移动一格；如果不为空，则插入位置开始向前移动一格，舍弃第一个元素，因为idel从小到大排序，小的可以舍弃
