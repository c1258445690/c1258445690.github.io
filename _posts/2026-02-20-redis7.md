## 10大数据类型
1. string
2. hash
3. list：双端队列
4. set：底层是intset或hashtable
5. sorted set：每个元素关联一个double类型的分数都set，根据分数排序。
6. geo
7. hyperloglog：基数统计，使用极少存储空间来统计大量不重复元素的个数
8. bitmap
9. bitfield：位域，支持范围操作比特位
10. stream：redis版MQ

### zset
+ zrange 是根据从0开始的索引查询，zrangeByScore是根据分数查询
+ zadd除了能添加，还能更新，其中GT表示只能更新为更大的分数，如果提供更小的分数则不更新，LT则相反，GT和LT不影响添加
+ zadd返回值是新增元素的个数，如果元素已存在则是修改，返回0，CH可以将返回值改为已修改元素的个数
+ zmpop: redis7新增，接收多个zset key，弹出第一个非空zset的n个元素, 默认1个，需要指定MIN还是MAX，弹出分数最小或最大的

### bitmap
+ 用于存储二值类型的数据，以字节为单位，底层是string，可以用strlen获取字节数

### hyperloglog
+ pfadd返回1表示hll的统计个数增加了，返回0表示不变，pfmerge合并2个hll。底层是string类型

### geo
+ 添加经纬度和值，底层是zset，经纬度会被编码成一个52位的整数，对应zset中的分数
+ geohash返回的是52位整数转换后的字符串，相同前缀的geohash表示在附近
+ geosearch: redis6新命令，用于替代georadius，查询附近的地点，可以根据成员名称或经纬度来查询，可以返回附近地点名称、距离、经纬度

### list
+ blpop: 一个客户端可以同时监听多个list，返回从左往右第一个不为空的list中的list名称和其中的一个元素；多个客户端可以同时监听一个list，获取元素的优先级取决于该命令请求的顺序，先请求的先获取到，再次请求优先级会重置，其它客户端会先消费到；当一个客户端同时监听多个空list，这些list通过事务或lua脚本同时写入元素时，客户端先获取到第一个写入的list的一个元素

### stream
+ 和list的区别：list每条消息只能保存一个值，而stream能保存多个键值对。
+ 和pubsub的区别，pubsub的消息不会持久化，消费者断线会丢失消息，无ack机制，消费出错无法重新消费。stream会持久化消息，有ack
+ xadd : 向stream添加<font style="color:#DF2A3F;">一条</font>内容为一个或多个键值对的消息，不存在则创建，如果指定了NOMKSTREAM，则不创建新的stream。*表示自动生成ID，ID由2部分组成，-分割，如果只指定一个整数n，生成的ID是n-0，ID必须单调递增。自动生成ID时，第一部分是时间戳，第2部分是系统时间小于等于当前stream最大时间戳时，递增的值，用于防止时钟回拨或者主从切换时主从系统时间不一致。支持添加消息的同时指定淘汰策略，相当于同时执行xadd和xtrim

#### xtrim
+ 淘汰老消息，可以通过剩余个数或ID来淘汰，返回淘汰了多少条消息
+ 根据剩余个数淘汰：剩下多少个最新的消息，其他老消息都淘汰，如果指定的个数大于stream消息总数，则不淘汰任何消息
+ 根据ID淘汰：小于该ID的消息都淘汰，等于该ID的不淘汰
+ 不指定和"="表示精确淘汰，
+ 近似淘汰:  "~"表示近似淘汰，淘汰的消息会小于等于精确淘汰的个数。近似淘汰时，还可以指定最大淘汰个数，默认是100*宏节点个数，0表示不限制最大淘汰个数。

#### xrange
+ 根据起始和结束ID范围查询stream，默认闭区间，-和+分别表示最小ID和最大ID
+ 可以通过COUNT指定查询最大消息数，返回原本结果集中的前n条消息
+ redis6.2新特性：通过"("和")"指定开区间，之前的版本需要手动将开区间边界增加后继续查询，来实现开区间
+ 因为ID分为2部分，如果只使用第1部分num，起始边界表示num-0，结束边界表示num-[第2部分最大值]
+ 开始和结束ID相同时，表示等值查询，如果只指定ID第一部分，则仍是范围查询

#### xdel
+ 根据ID删除消息，返回删除的消息个数，stream底层是radix tree存储，多条消息存储在同一区块中，删除是标记删除，当区块中所有消息都标记为删除时，才会真正删除区块
+ radix tree用于优化空间
+ [https://ivanzz1001.github.io/records/post/data-structure/2018/11/18/ds-radix-tree](https://ivanzz1001.github.io/records/post/data-structure/2018/11/18/ds-radix-tree)

#### xlen
+ 获取stream中消息个数，空stream和不存在的key都返回0，需要通过exists或type查询key是否存在，允许空stream存在，其他数据类型(list,set,map,zset)元素为空时，key会自动删除

#### xread
+ 查询大于指定ID的一个或多个steam中的消息，需要为每个stream指定起始ID，可以指定每个stream查询的最大条数COUNT
+ 一般用于迭代查询，从0开始，每次查询上一次返回的最大ID，返回nil时迭代结束
+ 和xrange的区别：xrange只能查询单个stream，可以指定结束ID
+ 只指定ID的第一部分n，等价于n-0
+ 可以通过BLOCK指定阻塞读取，超时时间的单位是ms，0表示无限等待，多个客户端阻塞读取时，新消息会分别发给所有阻塞读取的客户端，它们会收到同样的消息
+ 读取最新消息：将ID指定为"$"，注意：<font style="color:#DF2A3F;">只能第一次读取时使用$，后续读取不能用$，而是用上一次返回的最大消息ID，如果用$，会漏掉2次读取之间的所有消息</font>。

#### xgroup create
+ 创建消费者组，并且关联一个stream，指定消费者组起始消息ID，可以指定具体ID或$，后者表示从最新消息开始消费
+ 如果消费者组已经和该stream关联，报错：busy group。如果stream不存在，报错
+ 可以执行多次xgroup create为同一个消费者组关联多个stream。
+ $表示<font style="color:#DF2A3F;">创建该关联关系之后的所有消息</font>能被组中消费者消费到，而不是第一次xreadgroup之后的消息

#### xreadgroup
+ 消费者组消费，和xread的区别：1、多了消费组名称和消费者名称，同一消费组不会消费相同消息。2、也可以指定多个stream和多个起始ID，但<font style="color:#DF2A3F;">必须是消费组关联的stream的子集，真正的起始ID是消费者起始ID和消费组的起始ID的较小值</font>。比如消费组指定起始ID为n1，消费者起始ID为n2，起始ID是min(n1,n2)，n2=0表示从n1消费，n1是$且n2是>时，表示消费尚未消费的消息
+ $和>的区别：前者用于创建消费组，表示比最新消息ID更大的；后者用于xreadgroup，表示消费尚未消费的消息。前者不能用于循环消费，否则会丢失读取间隔中投递的消息
+ id=0用于重新消费未ack的消息，id=">"表示消费未投递过的消息

#### xack
+ 确认某条ID在某个stream的消息在消费组已被消费
+ xreadgroup消费的消息ID会保存到pending entry list(PEL)中，xack确认后消息ID才回从PEL删除
+ 未确认的消息可以通过xreadgroup指定id=0读取到

#### xpending
用于查询PEL，有2种形式：

1. 只指定消费组和stream：查询统计信息，PEL种id的个数，最小和最大id，以及其中的id对应的消费者名称和id个数。
2. 指定开始和结束id、COUNT：查询详细信息，每个未消费消息的id、对应消费者、距离上次投递的毫秒时间、投递次数。还可以只查询某个消费者未消费的消息。可以指定IDLE n，表示只查询距离上次消费的毫秒时间大于n毫秒的pending消息，用于查询长时间未投递的消息，然后使用XCLAIM
+ 支持迭代查询：每次使用上一次返回的最大ID和COUNT查一部分
+ 使用xreadgroup，指定ID=0可以重新消费PEL中的消息，并增加PEL中重新投递次数
+ 删除PEL：只能通过xack，xdel或xtrim只能删除消息内容，，PEL仍保存了消息ID、消息所属的消费者等信息，但通过xreadgroup查询时，只能查到ID，内容为nil，<font style="color:#DF2A3F;">因此xreadgroup消费消息时，需要判断消息体是否为空</font>。

#### xclaim
+ 用于修改PEL中消息所属的消费者，当某个消费者宕机无法恢复时，让其他消费者消费它的PEL消息
+ 需要指定消费组、stream名称、消费者名称、最小空闲时间、一个或多个消息ID，默认会返回空闲时间大于指定时间的消息ID和内容
+ 不在PEL或在PEL已删除的消息，不会返回
+ xclaim会重置空闲时间，递增投递次数，xreadgroup也会，如果xclaim指定了JUSTID，那么只返回消息ID，不递增投递次数，但会<font style="color:#DF2A3F;">重置空闲时间</font>
+ xclaim的其他选项用于AOF使用
+ 因为PEL是stream和消费组级别的，所以不能xclaim其他消费组的消息
+ 空闲时间的作用：多个消费者同时对一条消息xclaim，只有一个会成功，因为其中一个成功xclaim后重置了空闲时间，其他消费者设置了最小空闲时间，从而防止重复xclaim
+ 可以直接在每个消费者通过xpeding+xclaim来消费同组其他消费者空闲时间过长的消息，然后xack
+ 6.2新特性：xclaim已经删除的消息，不会返回该消息，并自动从PEL中删除

## rdb
+ rdb默认目录是当前目录，名称是dump.rdb，可以通过dir和dbfilename配置修改
+ fork系统调用：用于创建子进程，一次调用会产生父进程和子进程中的2次返回，父进程返回子进程pid，子进程返回0，可以通过判断返回值来区分父进程和子进程的执行逻辑。创建子进程时，子进程的虚拟地址空间指向父进程的物理空间，复用父进程的代码块和数据块。写时复制：只有当写入某个内存页时，才会复制对应内存页的物理空间
+ save配置：可以有多个配置对象，每个对象由2个字段组成，单位是秒数和脏数据的个数，**当距离上一次****<font style="color:#DF2A3F;">成功</font>****生成rdb的时间大于save配置的秒数，且脏数据的个数****<font style="color:#DF2A3F;">大于等于</font>****save配置的个数**时，触发生成rdb快照。如果配置了多个save，每次会遍历这些配置，如果有一个满足则触发
+ 如果上一次生成失败了，则尝试次数大于5次才会触发
+ <font style="color:#DF2A3F;">脏数据的个数</font>：不是key的个数，而是修改、新增、或删除的、值或键值对的个数(<font style="color:#DF2A3F;">值的个数</font>)，比如lpush多个值，这些值的总数加进脏数据个数
+ RDB快照中保存了魔数、内存数据集、结束符，如果此时有slave，还会保存复制ID和offset

### 什么时候会触发rdb快照
1. save配置条件满足时
2. 执行bgsave命令
3. save命令: 在主线程中生成rdb，一般不用
4. 主从复制
5. flushall: 如果配置了save，则立即触发一次rdb快照
6. shutdown：如果没有shutdown save，并且没有配置save，则不rdb，否则会rdb

注意：

+ flushdb只会将<font style="color:#DF2A3F;">删除的key的个数加到脏数据个数</font>中(server.dirty)，不一定会触发rdb，而是要满足save配置条件触发
+ 将save设置为空字符串""，表示关闭rdb快照自动生成，<font style="color:#DF2A3F;">bgsave和save不受影响仍能生成快照</font>。

### 优缺点
+ rdb好处：格式紧凑的单个文件，是整个数据集的全量快照，利于归档备份和恢复，比如可以每小时归档一次到高可用存储上，是二进制格式的，读取速度快，相比于AOF，启动时的加载速度更快
+ 缺点：因为save要满足一定条件才会触发，异常宕机时，会丢失上一次快照到当前的的数据。除了save命令、shutdown、flushall，其他方式都是通过fork子进程的方式bgsave，fork调用期间父进程不能处理客户端命令，另外如果数据集很大，并且生成rdb期间有大量命令写入，会触发大量写时复制，增加系统负载，可能出现卡顿

## AOF
+ aof默认关闭，通过appendonly yes开启
+ appenddirname：redis7新特性，之前的版本和rdb在同一目录，AOF文件所在目录，相对于dir目录，默认是appendonlydir
+ appendfsync：aof刷盘策略，always，everysec，no
+ appendfilename：AOF文件名称，默认是appendonly.aof，7之前的版本只有一个aof文件。redis7新特性：mutil aof，会生成3个aof文件，appendonly.aof.[num].base.rdb、appendonly.aof.[num].incr.aof、appendonly.aof.manifest

### AOF写入
+ 执行一条写命令后，先将该命令写入aof缓冲区中。写命令：和rdb判断脏数据的个数相同，执行该命令前后脏数据个数(server.dirty)增加了，将key修改为和原来相同的值，也会增加脏数据个数。
+ 写入aof：serverCron大循环中将缓冲区中的命令写入aof文件，写入后清空缓冲区
+ 刷盘：每秒刷盘：刷盘则是在单独的线程中进行，刷盘时会判断是否已有aof刷盘线程，确保只有一个该类型线程。always：同步刷盘，从缓冲区写入aof文件之后立即fsync

### AOF重写
+ 需要fork子进程重写，因为<font style="color:#DF2A3F;">重写是根据内存数据集的每个key生成对应redis命令，而不是原来的aof文件</font>。
+ 重写前，父进程先创建一个临时文件，将aof fd指向它，重写期间向该文件写入增量命令；重写子进程也创建一个临时文件，根据内存数据集的每个key生成对应redis命令，比如string对应set，list对应rpush，map对应hmset，set对应sadd，zset对应zadd，stream对应xadd、xgroup create
+ aof文件包括基础aof、增量aof和清单aof，基础aof是每次重写生成的，其中也是redis命令，是重写时的内存数据集快照；增量aof是客户端发送的写入命令转换为redis协议格式，包含重复数据；清单AOF记录了3种类型的文件：增量aof和清单aof，以及历史aof文件
+ 历史aof：重写以前的增量aof和清单aof，因为文件删除是异步线程中进行的，所以需要先保存历史文件信息，以便删除失败时恢复
+ 重写成功后，父进程重命名文件，将原来的基础aof文件重命名为历史基础aof文件，将重写子进程写的文件重命名为基础aof文件，将原来的增量aof文件重命名为历史增量aof文件，将临时增量aof重命名为增量aof，最后将新文件和历史文件名称加入清单文件，异步删除历史aof

## 事务
### multi exec discard
+ 事务是一组操作，要么全部成功，要么全部失败，并且根据隔离级别，执行过程中具备一定程度的排他性
+ redis事务是一个队列中，一次性、顺序性、排他性执行的一组命令，客户端先发送命令进入队列，提交时再一起顺序执行，因为redis是单线程的，事务命令执行期间不会有其他命令执行，其他命令在事务之后执行
+ <font style="color:#DF2A3F;">redis事务不能嵌套执行</font>，事务通过exec或discard结束前不能再次执行multi
+ 执行multi命令实际上就是开启客户端的事务标志位，执行命令时会先判断客户端是否处于事务状态，如果是，则先不执行该命令，而是将它写入客户端对应的队列中
+ exec执行之前，入队的命令报错，会导致整个事务不执行，比如语法错误，内存不足等，返回exec abort；exec执行的报错只有对应报错的命令不执行(部分成功)，因此redis事务并没有做到完全的原子性, 未执行的命令返回nil
+ 事务不会执行阻塞命令，比如blpop，如果执行了，阻塞命令不执行，返回nil，但不会影响事务中其他非阻塞命令的执行
+ exec返回值：依次对应每条事务命令的返回值

### watch
+ watch：用来对key加乐观锁，只有当key在watch和mutil期间未被修改，事务才能执行成功，否则整个事务不执行。使用场景：需要将一个key查出来做一些修改再保存，不希望修改期间其他客户端修改
+ watch不能在事务中执行，可以watch多个key，可以监听当前Redis库不存在的key。
+ 什么时候整个事务不执行：1、执行discard。2、exec执行前，命令入队时报错。3、存在任一个watch的key被修改或者过期了
+ redis不支持事务回滚，discard只是不执行进入队列的命令

## 管道
作用：减少命令的往返时间，通常redis命令的处理包括发送命令、执行命令、返回响应，当需要一次性发送大量命令时，每条命令都需要一次发送和响应，网络通信时间占用了较大比例的总时间。管道将这些命令一次性打包发送，redis执行之后，再打包返回执行结果，

1. 批量发送提高了网络IO的效率，单条命令的方式，每条命令都需要read和write系统调用，而管道总共只需要一次
2. 单条命令的方式通常需要等待上一个命令的执行结果，再执行下一条命令，而管道一次性批量接受所有命令的响应
+ redis服务端会缓存管道的命令的执行结果，因此一次管道执行的命令数不宜过多，否则占用过多服务端内存，可以每次发送10k命令(如果每个命令10B，就是100条命令)
+ redis-cli --pipe从标准输入读取redis命令
+ 和事务的区别：只会依次执行，不保证原子性
+ 和原生批量命令的区别：原生批量命令只支持同种类型的key，管道支持不同类型的key
+ 和lua脚本的区别：1、lua脚本能保证原子性，管道不保证原子性。2、如果需要上一条命令的执行结果，管道实现不了，只能脚本。3、脚本可以不返回中间结果，管道会返回每条命令的结果。
+ 管道其实是由客户端实现的，服务端感知不到，服务端仍然发送一条命令处理一条，返回执行结果，客户端做了2件事情：<font style="color:#DF2A3F;">1、发送时安装一定大小的批次发送，批次大小可以自定义。2、不阻塞等待每条命令的执行结果，而是将一个批次的命令发送后，再一并等待</font>。

### 客户端实现
+ redisTemplate有2个api使用管道：1、execute方法，pipeline参数设为true。2、executePipelined方法。这2种方法的区别：前者获取不到管道中每条命令的执行结果，只适用于只写命令的管道；后者的返回值是List，对应每条命令的执行结果
+ <font style="color:#DF2A3F;">executePipelined的RedisCallback必须返回null，否则报错</font>。
+ executePipelined其实内部还是调用了execute，通过closePipeline得到命令的执行结果
+ redisTemplate.execute的第2个参数为true时，RedisCallback中获取到的连接是未经过代理的对象，比如LettuceConnection，为false时是经过代理的对象，CloseSuppressingInvocationHandler拦截了close方法，防止业务代码误关闭连接
+ openPipeline：执行管道命令前调用，纯客户端操作，初始化结果列表和刷新策略，将连接标记为管道，以便后续使用专有连接。closePipeline：等待所有命令的执行结果，将结果保存到List返回
+ 刷新策略：默认每条命令发送后都会冲刷网络缓冲区，可以配置成关闭管道时冲刷，或者发送固定命令数后冲刷
+ <font style="color:#DF2A3F;">Lettuce中，非管道和非事务命令，底层公用一个连接；管道和事务命令使用专有连接。</font>因为管道和事务较耗时，为了避免影响其他线程执行命令。
+ 管道和非管道命令的区别：1、前者使用专有连接，后者共享连接(对lettuce而言)；2、前者发送命令后不阻塞等待执行结果，而是将命令执行结果的Future对象保存起来，closePipeline时再等待执行结果，后者执行每条命令阻塞等待执行结果

## pubsub
+ subscribe可以订阅多个通道，返回值由3部分组成：1、消息类型，subscribe命令对应的类型是subscribe，unsubscribe是unsubscribe，接收到消息是message。2、通道名称，表示订阅或接收到消息时的通道名称。3、subscribe或unsubscribe是当前订阅的通道数，接收到消息时是消息体
+ <font style="color:#DF2A3F;">订阅了通道的连接只能发送固定的命令(ping、(P|S)SUBSCRIBE / (P|S)UNSUBSCRIBE,quit,reset)</font>，RESP3支持所有命令
+ reset:1、退出登录。2、如果处于事务中，discard事务。3、unwatch所有正在watch的key。4、退出订阅状态。5、select 0。
+ quit断开连接
+ 发布订阅和客户端连接选择的redis库无关，pub和sub的客户端选择了不同库，仍能接收到消息
+ publish的返回值是接收到该消息的客户端数量
+ psubscribe支持根据"*"通配符订阅通道，对应punsubscribe，取消订阅也必须使用同名带通配符的通道。psubscribe订阅的消息由4部分组成：1、消息类型，pubscribe。2、带通配符的通道名称。3、实际匹配到的通道名称。4、消息体

## 主从
+ replicaof：slave配置文件和命令中设置主节点。replicaof no one: 将从节点变为独立的主节点，旧版本slaveof
+ 从节点是只读的
+ 从节点过多会导致master带宽占用过高，从而影响客户端命令的发送和接受，解决方法：多级同步，slave向其他slave同步
+ 主节点宕机，从节点不会自动切换，而是等待主节点上线，期间集群只能处理只读命令
+ 从节点首次连接：会向master发送psync命令，全量复制，会清空原来的数据
+ 全量同步：主节点收到psync命令后会触发生成RDB，同时将所有接收到的用于修改数据集的命令缓存起来，主节点完成RDB后，将RDB快照文件和所有缓存的命令发送到slave。slave接收到数据后，将其存盘并加载到内存
+ 心跳：master向slave发送PING，间隔repl-ping-replica-period，默认10s
+ 增量复制：master将新的收集到的修改命令依次发送给slave
+ 断点续传：master会检查backlog中的offset，master和slave都会保存一个复制的offset和masterId，master只会把已经复制的offset后面的数据复制给slave



+ 过期和内存淘汰的key也是通过master同步给slave进行删除
+ 断点续传失败时，会全量同步
+ master向slave同步，以及slave确认命令已同步都是异步的
+ master可以不开启RDB快照只开启AOF，slave开启RDB，这样减少了master的磁盘IO，但是重启时要注意，如果master没有开启任何持久化配置，重启时会丢失所有的数据，然后slave也会同步清空。
+ master的数据集会对应一个随机数的复制ID和偏移量，偏移量是数据集的字节数，slave端也会保存复制ID和偏移量，PSYNC时，slave将复制ID和偏移量发送给master，master通过复制ID找到偏移量，判断slave的同步进度，如果master的backlog缓冲区包含slave所需的数据，则进行增量同步，否则全量同步。
+ 复制ID的作用：<font style="color:#DF2A3F;">主从切换时避免全量复制</font>。当节点第一次成为master，或者从slave提升为master时，会重新生成复制ID，slave连接master时，会复制master的复制ID作为自己的，因此复制ID代表了一届master的数据版本，一个master和它的所有slave的复制ID相同。当slave提升为master时，其他slave因为复制ID和它相同，只需要进行增量复制。
+ 当多台slave同时连master时，master只进行一次RDB。
+ SYNC命令不再使用，而是PSYNC
+ repl-diskless-sync：设置master无rdb同步
+ masterauth：从节点的配置，主节点的密码
+ min-replicas-to-write  <number of replicas>，min-replicas-max-lag <number of seconds>：主节点可以配置有多少个从节点存活时，才接受写命令，一定程度上可以确保数据安全，但由于同步是异步的，主节点返回写入成功时，从节点不一定写入成功。前者是从节点个数，后者是心跳时间距离当前时间的秒数小于配置值，表示存活
+ replica-serve-stale-data：从节点配置，默认为true，master下线时，从节点仍能响应读命令，为no时，从节点拒绝客户端读数据的命令，返回MASTERDOWN Link with MASTER is down and replica-serve-stale-data is set to 'no'.
+ repl-diskless-sync：默认yes，为false时，全量同步时会fork子进程，子进程在磁盘上生成RDB文件，父进程将它发送给slave；为true时，全量同步时fork子进程，子进程不在磁盘上生成RDB，而是直接从内存中发送给slave。前者在生成RDB文件期间，连接进来的slave会进入队列等待，生成完成后将同一个RDB文件发送给队列中的slave；后者因为传输一旦开始，新的slave进入队列等待，传输完成后再处理队列中的slave，所以会等待几秒，以便并发处理。diskless适合慢磁盘高带宽的场景
+ repl-diskless-sync-delay：diskless同步，等待时间最长的slave大于等于该时间时
+ repl-diskless-sync-max-replicas：diskless同步，额外等待的过程中，如果等待的slave个数达到配置值时，不再等待，提前开始同步，默认0，不启用
+ repl-ping-replica-period：master向slave发送ping的间隔，单位秒
+ repl-backlog-size：master的复制缓冲区大小，越大则容忍slave断联的时间越长，第一个slave连接时，才会创建复制缓冲区
+ repl-backlog-ttl：当没有slave时，复制缓冲区的超时时间，超时释放，为0表示永不超时，默认1h
+ repl-diskless-load: slave diskless同步，slave接收到同步数据后，是否先保存到<font style="color:#DF2A3F;">临时rdb</font>文件再加载到内存数据集，默认disabled，先加载磁盘RDB，可能会导致全量加载慢；swapdb：直接解析sokcet中的RDB数据，解析过程中原有的数据集和正在加载的共存，可能会导致OOM；on-empty-db：原有的内存数据集为空时，才直接加载socket，否则先加载到磁盘
+ rdb-del-sync-files：是否删除从master同步过来的RDB文件，默认no不删除，为yes时，同步完成后并且rdb和aof都关闭时会自动删除

### master同步
+ slave client状态replstate流程：SLAVE_STATE_WAIT_BGSAVE_START->SLAVE_STATE_WAIT_BGSAVE_END，当发送psyn命令时，先处于等待bgsave状态，当没有正在运行的子进程时，等待结束，可以fork RDB子进程
+ 对于socket RDB子进程，不会直接通过子进程将RDB数据发送给slave，而是通过管道发送给父进程，然后父进程发送给slave
+ 延时批量处理：配置了repl_diskless_sync和repl-diskless-sync-delay时，实际上不会立即处理slave的psync请求，而是放到slaves中，然后在replicationCron中统一处理,它是定时1s执行的，从而实现一定程度的匹配执行psync
+ slaveRDB同步完成后，会发送replconf ack命令，master将client加到待写列表，RDB同步期间的命令会写到复制缓冲区，随后主事件轮询中遍历待写列表，复制缓冲区的命令写到socket
+ 复制缓冲区由复制缓冲块列表和索引组成，缓冲块中保存了命令数据，索引用来根据偏移量快速查找缓冲块
+ 全量同步分为RDB同步和命令同步，增量同步的逻辑和命令同步相同

### 增量同步
+ slave的复制ID和偏移量都会持久化到RDB，重启时如果能从RDB中加载到它们，则进行尝试进行增量复制，发送psync，master收到后，判断如果复制ID和当前复制ID相同，并且偏移量在复制缓冲区范围内，则将偏移量对应的缓冲块加到slave output buffer，随后写到socket
+ 如何触发增量同步：1主1从，停止slave，向master写入命令，启动slave

#### <font style="color:#DF2A3F;">触发增量同步的场景</font>
1. slave宕机重启，slave从rdb中加载复制ID和偏移量到<font style="color:#DF2A3F;">cached_master</font>，然后尝试psync增量同步(cache_master是不带连接的master client)
2. 故障切换，slave切换为master之前，会备份原来的复制ID和偏移量，再生成新的复制ID，当它晋升为master后，接收到其他slave的psyn命令时，slave的复制ID和备份复制ID相等，并且复制偏移量小于备份偏移量second_replid_offset，且在复制缓冲区范围内，则满足增量复制的条件，进行增量复制
3. 哨兵故障切换，原master上线时，使用rdb中的复制ID和偏移量创建cache_master，连接新master时，会尝试使用cache_master的复制ID和偏移量进行增量复制

## 哨兵
+ 哨兵可以同时监控多个master，通过配置多个sentinel monitor [master name] [ip] [port] [quorum]，master name的作用：配置分为2类，全局配置和master配置，master配置需要通过master name来指定在哪个master生效，master配置格式：sentinel <option_name> <master_name> <option_value>
+ 哨兵启动使用专门的执行程序和配置：redis-sentinel sentinel.conf， 相当于redis-server   sentinel.conf  --sentinel，默认端口26379
+ 启动后会修改redis server和哨兵的配置: 哨兵只需要配master名称和ip端口号，会自动发现slave, 并动态添加到哨兵的配置中，
+ 主观下线：单台sentinel ping master超时；客观下线：quorum个哨兵认为master主观下线。客观下线时，哨兵会内部选举出一个leader(raft)，由leader哨兵选择新master（failover）,判断客观下线不一定需要过半sentinel节点，quorum个即可，而主从切换需要过半节点在线
+ 选master：配置的优先级>repli offset>run id，sentinel leader对新master执行replicaof no one，其他slave执行replicaof 新master
+ 主从切换时，会有一段时间出现master不可用
+ sentinel需要客户端支持
+ down-after-milliseconds: 主观下线超时时间
+ parallel-syncs ：主从切换后，多少台slave可以同时连master。数值越小，重新同步的时间越长，但如果配的过大，slave在同步期间会有一段时间不可用，导致集群可用性变差，默认1
+ min-replicas-to-write，min-replicas-max-lag：master的配置，至少有几个slave在线时，master才接受写请求。目的是防止client、master和slave网络分区时，slave中选出新master，而client仍向旧master写数据，分区恢复后，旧master变为slave，清空自身数据同步新master，导致client写成功的数据丢失。副作用：当所有slave宕机时，master不能写入数据
+ sentinel通过pub sub来向客户端实时推送master故障切换信息
+ sentinel ckquorum <master name>: 检查哨兵是否满足quorum个在线
+ sentinel failover <master name>: 手动切换master，不需要其他哨兵同意
+ <font style="color:#DF2A3F;">SENTINEL GET-MASTER-ADDR-BY-NAME <master name></font>：获取当前master ip 端口
+ 配置动态修改：监控配置：SENTINEL MONITOR <name> <ip> <port> <quorum>。停止监控：SENTINEL REMOVE <name>。master配置：SENTINEL SET <name> [<option> <value> ...]。全局配置：sentinel config set ...。<font style="color:#DF2A3F;">需要对所有哨兵都执行修改命令，不会自动同步动态修改的配置。动态修改后会自动保存到哨兵配置文件</font>。
+ 读取master所有配置：SENTINEL MASTER <master name> 
+ 全局配置：哨兵用户名密码登
+ 移除哨兵：哨兵之间会相互发现，并保存到内部状态中，停止哨兵进程后，还需要在其他哨兵节点执行sentinel reset ，让它们清空内部状态重新发现，然后在每台哨兵执行sentinel master查看是否成功。移除slave也一样，停止slave进程后需要执行sentinel reset

### 故障切换（主从切换）
+ master下线后，等待down-after-milliseconds时候后才会检测到sdown，触发failover
+ SENTINEL IS-MASTER-DOWN-BY-ADDR <master ip> <master port> <epoch> <runid>：向哨兵根据ip端口号查询master是否sdown，如果ip端口是master并且sdown，第一个值返回1，否则0. 如果runid为*，最后2个参数忽略。哨兵通过定时向其他哨兵发送该命令，来实现sdown查询，更新master.slaves.flags为SRI_MASTER_DOWN
+ **判断sdown**：当前时间-act_ping_time是否大于 down-after-milliseconds，act_ping_time是最近一次发送ping且尚未接收到pong的时间戳，当接收到pong时，会设为0
+ **判断odown**：master的SRI_S_DOWN开关+master.sentinels的SRI_MASTER_DOWN开关开启总数>=配置的quorum数，如果odown，将master的SRI_O_DOWN开关开启，记录+odown日志，发布消息
+ **leader选举**：第一个检测到master odown的哨兵(候选节点)，进入故障切换状态，开始投票选举leader，先自增epoch，给自己投票，然后向其他哨兵发送投票请求。其他哨兵接收到投票请求后，发现epoch大于自己的epoch，则同意投票请求，也发起投票请求投该候选节点。候选节点接收到过半请求后，成为leader，开始晋升slave
+ leader选举时，当其他哨兵接收到epoch大于自己的投票请求后，会重置故障切换超时时间，随机1~100s内不会进入故障切换状态(成为候选节点)，从而避免也发起投票请求
+ 查询其他哨兵是否认为master sdown和投票请求都是通过SENTINEL IS-MASTER-DOWN-BY-ADDR 命令实现，查询master sdown阶段，runid=*，忽略最后2个参数
+ **选择待晋升的slave**：首先去掉配置replica-priority等于0的，然后选择replica-priority最小的，replica-priority相同时，选择复制偏移量最大的，复制偏移量相同时，选择runid自然排序靠前的。
+ 复制偏移量：slave从复制缓冲区复制时，递增该值，全量复制时，同步rdb之前slave会从master同步它。master会在rdb和aof中持久化它，从而保持递增
+ **向待晋升的slave发送replicaof no one命令**、config rewrite、client kill type normal/pubsub 命令，在事务中执行这3个命令，候选slave接收到replicaof no one时，会释放和原master的连接、备份原复制ID和偏移量，生成新的复制ID，以便后续slave判断是否可以增量同步。候选slave更新replicaof 配置
+ client kill type normal/master/replica/pubsub: 类似于mysql的kill，client list 类似于mysql 的show processlist，查询对应实例的客户端连接，addr客户端地址，laddr本地地址，type：normal表示哨兵和用户的非订阅命令的客户端，pubsub表示subscribe客户端。
+ **等待slave切换成master**：当通过info命令查询到slave的角色变为master时，向其他slave发送replicaof <master ip> <master port>、config rewrite、client kill，按批次发送，每批parallel-syncs个slave，其他slave收到后，断开和原master的连接，和新的master建立连接并开始同步。更新replicaof配置
+ **哨兵之间同步新master信息**：通过hello消息同步，如果config_epoch大于自身，并且消息中的master地址变更时，reset master，包括哨兵更新master地址，原来master的地址加到slaves，重置故障切换状态、选票、ping超时时间、选举超时时间
+ **原master切换为slave(旧主拜服)**：原master上线后，哨兵通过info命令判断它是原来的master，对原来的master发送replicaof 命令，原master接受后，使用rdb中的复制ID和偏移量创建cache_master，连接新master时，会尝试使用cache_master的复制ID和偏移量进行增量复制

**SENTINEL IS-MASTER-DOWN-BY-ADDR <master ip> <master port> <epoch> <runid>**

+ 做2件事情：1、查询其他哨兵是否认为master sdown。2、选举哨兵中的leader。
+ 4个参数：前2个是master ip端口号，后2个当正在故障切换时，发送投票请求，runid是投给哪个哨兵的myid，epoch用于是否同意投票
+ 3个返回值：第一个为1表示master sdown，后面2个用于选举，对端实例投给谁的myid和它的leader_epoch

**sentinelState.current_epoch**

+ epoch计数器，每次故障切换加1
+ 启动时根据sentinel config-epoch配置赋值，修改该值后会保存到配置文件
+ hello消息中包含该值(第3个参数)，根据hello消息将哨兵实例的该值同步为它们的最大值

**hello消息**

+ 哨兵定时向3种类型的实例发布和订阅hello消息，8个字段分别是哨兵ip,port,哨兵myid,current_epoch,master name,master ip,port,config_epoch
+ 先根据master name从masters中查找master实例，然后做以下任务
1. 哨兵之间服务发现
2. 如果current_epoch大于自身，更新current_epoch
3. 哨兵之间同步master信息：config_epoch大于自身，并且消息中的master地址变更时，哨兵更新master信息,reset master

## 集群
+ 优点：1、写操作高可用：哨兵是单台master，写操作存在单点故障问题。2、单节点内存有限。3、redis单实例是单线程的，只用到了1个核，集群发挥了多核优势
+ 为什么不直接对redis实例数取模，来计算key所在的实例（哈希取余分区）：不方便扩缩容，扩缩容时，需要重新计算所有key对hash值并取模，重新为所有key分配实例。
+ 一致性哈希分片：在key和实例之间加了一个中间层：哈希环，哈希环相当于环形数组，分为2^n个槽位，编号为0～2^n-1，key的hash值对2^n取余，得到key在哈希环的槽位，实例均匀分布在哈希环上，得到key的槽位后，顺时针查找到的第一个实例就是key对应的实例。缩容时，缩容实例a逆时针方向到第一个实例的key迁移到a顺时针方向第一个实例。优点：扩缩容不需要全量数据重新计算hash值。缺点：经过几轮扩缩容后，实例不再均匀分布，从而导致数据倾斜。
+ 哈希槽分片：在key和实例之间加了一个中间层：哈希槽，哈希槽相当于线性数组，分为2^n个槽位，编号为0～2^n-1，key的hash值对2^n取余，得到key在哈希槽的槽位，槽位在实例的分布可以自定义，比如每个实例分配相等数量槽位，因此扩缩容能避免数据倾斜
+ 为什么redis哈希槽位个数是16384：CRC16的结果有16位，支持65536个槽位，但哈希槽会通过心跳包在集群间同步，槽位数过多会导致心跳包过大。而redis官方建议集群master个数不超过1000，槽位太小会导致数据倾斜

```plain
hash(key) mod 3
k1 v1 -> 1
k2 v2 -> 2
k3 v3 -> 0

hash(key) mod 4
k1 v1 -> 1
k2 v2 -> 2
k3 v3 -> 3

n -> m m>n
0~n 

hash(key) mod 16384  n
n -> m
```

+ cluster failover：手动主从切换命令，在待晋升的slave上执行该命令，和自动故障切换不同，slave收到该命令后，master会停止客户端写入请求，然后slave完全同步master的数据，切换为master，继续执行客户端写入请求
+ cluster countkeysinslot <slot num>：
+ cluster keyslot <key>：获取key对应的槽位
+ 集群不保证强一致性：1、主从之间是异步复制，客户端接收到master写入成功的ack后，master可能还没有同步给slave就宕机了，写入master的数据丢失。2、发生网络分区，处于少数节点分区的客户端在node_timeout时间窗口内写入的数据会丢失。
+ node_timeout：master联系不到大多数节点的时间超过该时间时，会拒绝写入请求，并且会触发故障切换
+ pub/sub：可以在集群任意节点上pub/sub，内部会转发消息

### hash tags
+ key中包含一对大括号，存储该key时，会使用大括号内的子串来计算槽位
+ 如果有多对大括号，会使用第一个"{"和第一个"}"内的子串，比如foo{bar}{baz}, 使用bar，foo{{bar}}，使用{bar
+ 如果大括号内为空，会使用整个key来计算。使用场景：1、key本身就有大括号。2、key是二进制的，内容不可控。可以在key前加{}，来使用整个key（包括空大括号）计算槽位
+ 集群模式不支持多个key的批量操作（mset,mget，事务，lua）操作不同槽位的key，如果要操作多个key，必须通过{}设置相同槽位，不支持批量操作不同槽位的key是因为redis集群节点不转发命令请求，而是返回key所在的槽位的实例地址。

```plain
unsigned int keyHashSlot(char *key, int keylen)//根据key计算hash槽位，取大括号内的或者整个key，然后调用crc16(key) & 0x3FFF
```

### 搭建
+ 2个端口：除了客户端使用的端口，集群节点间使用另一个端口通信，端口号=10000+客户端使用的端口
+ cluster-enabled <yes/no>：集群模式必须开启该配置
+ cluster-config-file <filename>：集群配置文件名称，每个集群节点都要配置单独的文件，注意同一台机器上的名称不能重复，集群配置文件由redis生成，里面保存了集群其它redis实例的ip,port等信息
+ cluster-node-timeout <milliseconds>: 节点超时时间，master超过该时间联系不到大多数节点时，会触发故障切换，并且该master拒绝读写请求
+ cluster-slave-validity-factor <factor>: 该配置用来提高数据一致性，减少可用性。是slave的配置，故障切换时，超过node-timeout*factor时间未连上master的slave，不会晋升master。如果没有满足条件的slave，则整个集群不可用，直到原宕机的master上线。默认是0，表示不判断slave未连上的超时时间。
+ cluster-require-full-coverage <yes/no>: 默认yes。16384个槽位中存在未被集群节点覆盖的槽位时，集群拒绝写入请求。
+ cluster-allow-reads-when-down <yes/no>: 默认no。集群不可用(cluster.state=CLUSTER_FAIL)时，其中存活的节点是否提供读取服务，默认拒绝读取。如果设为true，集群不可用时（比如槽位没有完全覆盖、网络分区中的少数分区的节点），存活的节点仍响应读请求，可能会导致读到脏数据
+ 创建集群：1、cluster-enabled 设为yes，配置cluster-config-file。2、通过redis-cli --cluster create创建集群，--cluster-replicas表示每个master对应多少个slave
+ 重分片：为指定节点分配槽位，可以是新添加的节点，也可以是已有槽位的节点
+ 集群扩容：包括添加master节点和slave节点，默认新增节点为master，slave节点需要通过--cluster-slave指定，slave会随机分配给slave最少的master,也可以通过cluster-master-id指定分配给哪个master的nodeId。还可以通过在已经分配master的slave上执行cluster replicate <master node id>命令，来修改对应的master
+ 缩容：通过node id来删除，如果删除的是master节点，需要先重分片，使该master不占用任何槽位，或者通过cluster failover将master切换为slave，再del-node删除。但如果要删除没有slave的master，只能重分片再删除。del-node的删除方式必须所有节点都在线，因为该命令需要跟所有节点通信，如果有节点不在线，可以使用通过call命令向集群每台存活节点只想cluster forget来删除。
+ redis-cli --cluster call 127.0.0.1:7000 <command>: 向集群每个节点执行命令，比如执行config set
+ del-node实际上包括了对除了待删除的其它节点执行cluster forget、对待删除的节点执行cluster reset soft
+ cluster forget <node id>：比如在a节点上执行，a节点从本地节点表移除node id	

**cluster reset**

+ 分为cluster reset soft和cluster reset hard，会清空自身数据集和槽位映射关系、集群节点信息，master节点需要先迁移掉槽位后才能reset
+ 执行del-node时，最后一步就是在待删除掉节点上执行reset
+ reset hard：会重新生成node id, 将currentEpoch和configEpoch置为0

### 节点状态、服务发现
+ 节点状态保存在节点配置文件中，状态分为全局状态和本地状态，全局状态（比如node id、ip、port、master还是slave、slave的master node id）会在集群节点间同步，本地状态不会同步
+ node id：集群模式的节点第一次启动时生成，不会改变
+ 状态：node id、ip、port、master还是slave、slave的master node id、最近一次接收到ping的时间戳、最近一次接收到pong的时间戳、当前节点分配到的槽位范围、config epoch、连接状态(connnected)。cluster nodes命令获取的就是节点状态
+ 节点间通信的端口默认是10000+port，也可以通过cluster-port配置
+ N个节点的集群，每个节点都会和其它N-1个节点建立TCP连接，因此每个节点有N-1个出站连接和N-1个入站连接，使用gossip协议更新状态，防止消息过多
+ 节点服务发现：只能通过2种方式：1、在待加入集群的节点上执行CLUSTER MEET <ip:port>命令，ip:port是集群中任意一台节点，执行该命令后，待加入的节点会发送meet消息给ip:port节点，接收者收到meet消息后，会将发送者加入到集群中，保存到当前节点状态中。2、通过gossip协议服务发现，比如A发现了B，B发现了C，那么B会将C的信息发送给A，A就会发现C。因此加入新节点时，只需要cluster meet其中任意一台节点即可，集群会通过gossip自动发现
+ 集群中的节点收到key操作命令时，会先检查是否key的槽位是否被当前节点覆盖，如果没有，返回-MOVED error，error信息中包含key的槽位、所在的节点IP、port，客户端可以在启动时或收到MOVED error时，执行cluster shards命令查询集群所有槽位和节点对应关系，从而避免服务端返回MOVED
+ CLUSTER GETKEYSINSLOT <slot num> <count>: 根据槽位号获取其中的key
+ CLUSTER COUNTKEYSINSLOT <slot num>： 根据槽位号统计其中的key的个数

### 槽位迁移
+ 和重分片的区别：后者只能根据槽位数来迁移，用于给扩容时新添加的master分配槽位、缩容时删除master节点前移除槽位，前者可以根据某个具体的槽位号来迁移。重分片支持迁移有数据的槽位

假如将7000上的槽位449迁移到7001，步骤：

1. CLUSTER SETSLOT 449 importing 9a59ef50837143a096f86d5a05c1d7af043116b9，在目标节点执行CLUSTER SETSLOT importing命令，目标节点的449槽位进入importing状态，这时对目标节点importing状态的槽位的查询，会MOVED重定向到源节点。只有<font style="color:#DF2A3F;">ASKING</font>查询会直接执行。
2. CLUSTER SETSLOT 449 migrating 074bcd5ba8c25f5b17c929002887d4433331f7b9，在源节点上执行CLUSTER SETSLOT migrating命令，源节点的449槽位进入migrating状态，这时对源节点migrating状态槽位的查询，存在的key直接返回，不存在的key会<font style="color:#DF2A3F;">ASK重定向</font>到目标节点，客户端收到ASK后，会在目标节点上执行ASKING，然后在执行真正的命令。使新增的key会保存到目标节点。
3. CLUSTER GETKEYSINSLOT 449 10，在源节点上执行该命令，查询槽位中的key
4. MIGRATE 10.8.0.7 7001 "" 0 3000 keys k2, 根据槽位中的key，在源节点上执行该命令，触发迁移操作，源节点接收到migrate命令后，会执行dump生成key对应的键值对序列化数据，然后向目标节点发送restore命令，发送给目标节点，然后del删除key，这些动作会阻塞2个节点，从而确保原子性
5. CLUSTER SETSLOT 449 NODE 074bcd5ba8c25f5b17c929002887d4433331f7b9，当源节点槽位所有key都迁移完成，为空时，在所有master节点执行该命令，更新槽位信息
+ <font style="color:#DF2A3F;">ASK重定向和MOVED重定向的区别</font>：2者都表示当前节点不存在该key，需要到重定向节点获取。客户端接收到MOVED重定向时，表示槽位永久映射到重定向节点，可以更新本地的槽位到节点映射关系，而ASK重定向只需要本次访问需要重定向，不能更新映射关系，并且向重定向节点发送命令前，需要先发送ASKING命令。
+ ASKING命令是一次性的

+ 心跳：每个节点在node_timeout/2时间内给其它所有节点发一次ping消息。实现方法：每次轮询遍历每个节点，选择其中没有acting ping，并且最近一次收到pong的时间距离当前时间超过node_timeout/2的，发送ping
+ 心跳包由2部分组成：1、和其它类型的包一样的公共的头部：node id, master/slave flag, currentEpoch,configEpoch, 发送者的slot bitmap, port, cluster-port, 集群down/up, master node id(如果是slave)。2、gossip段：包含了发送者知道的节点的随机子集的信息，每个节点信息：node id, ip, port, node flags。gossip段没有包含所有已知节点的信息，可用于服务发现和故障检测。
+ 槽位到节点的映射信息通过心跳和update消息更新，当节点收到其它节点的心跳时，如果心跳的configEpoch大于自身，则更新自身的槽位映射和映射entry对应的configEpoch，如果小于自身，则发送update消息给对端节点，对端节点更新槽位信息。槽位映射表的每条entry都对应一个configEpoch

```c
void clusterCron(void) {
    //cluster cron每秒执行10次，因此下面的if每秒执行一次
    if (!(iteration % 10)) {
        int j;
        //随机选择5个没有acting ping的节点，选择其中一个最近一次收到pong时间距离当前时间最久的，发送ping
        for (j = 0; j < 5; j++) {
            de = dictGetRandomKey(server.cluster->nodes);
            clusterNode *this = dictGetVal(de);
            /* Don't ping nodes disconnected or with a ping currently active. */
            if (this->link == NULL || this->ping_sent != 0) continue;
            if (this->flags & (CLUSTER_NODE_MYSELF|CLUSTER_NODE_HANDSHAKE))
                continue;
            if (min_pong_node == NULL || min_pong > this->pong_received) {
                min_pong_node = this;
                min_pong = this->pong_received;
            }
        }
        if (min_pong_node) {
            serverLog(LL_DEBUG,"Pinging node %.40s", min_pong_node->name);
            clusterSendPing(min_pong_node->link, CLUSTERMSG_TYPE_PING);
        }
    }
    //遍历每个节点，选择其中没有acting ping，并且最近一次收到pong的时间距离当前时间超过node_timeout/2的，发送ping
    while((de = dictNext(di)) != NULL) {
        mstime_t ping_interval = server.cluster_ping_interval ? 
                server.cluster_ping_interval : server.cluster_node_timeout/2;
        if (node->link &&
            node->ping_sent == 0 &&
            (now - node->pong_received) > ping_interval)
        {
            clusterSendPing(node->link, CLUSTERMSG_TYPE_PING);
            continue;
        }
    }
}
```

#### 故障检测
+ PFAIL：每个节点（比如a）向其它节点发送ping心跳包，当某个节点b在NODE_TIMEOUT时间内未回复PONG，则a将b的PFAIL flag开启，认为b主观下线
+ FA IL： 心跳包的gossip段包含了已知节点的PFAIL flag，当a收到过半分配了槽位的master节点认为b pfail，并且该状态持续了2*node_timeout时(根据收到pfail心跳的时间戳是否大于2*node_timeout)，a将b设为fail flag，向集群其它节点发送FAIL消息，注意FAIL消息不是心跳包，而是专门的消息，其它节点收到后，不管是否认为b pfail，都会将b设为fail。
+ 故障检测没有主从之分
+ 统计pfail票数时，只统计<font style="color:#DF2A3F;">分配了槽位的master节点</font>，因此一般建议master个数为奇数个。
+ 当检测到master节点fail时，会导致槽位未完全覆盖，集群不可用(cluster.state==CLUSTER_FAIL)，从而<font style="color:#DF2A3F;">会有短暂时间节点不能处理请求</font>，返回-CLUSTERDOWN The cluster is down。

#### 选举master
+ 由slave检测和发起自动故障切换，触发主从切换的条件：1、slave故障检测到master节点FAIL客观下线。2、客观下线的master分配了槽位。3、slave和master断线时间不超过cluster-slave-validity-factor * node_timeout。4、cluster-replica-no-failover配置为no
+ 如果master没有分配槽位，手动故障切换也不生效。slave没有检测到master fail或开启了cluster-replica-no-failover时，可以手动故障切换
+ <font style="color:#DF2A3F;">手动故障切换，只有当master没有分配槽位时切换不成功</font>，否则一定会切换。
+ 流程：该master的slave会递增currentEpoch，然后<font style="color:#DF2A3F;">向集群的master节点发送投票</font>请求（FAILOVER_AUTH_REQUEST包），master节点在NODE_TIMEOUT * 2时间窗口内，只会给同一master的1个slave投票，防止同一的多个slave晋升为master。如果同意投票，则回复FAILOVER_AUTH_ACK
+ 所有节点currentEpoch最终一致，同步为最大的值，--cluster create初始化集群时，currentEpoch从1开始依次递增。比如7000~7005的值是1~6，服务发现后，通过心跳包同步为最大值6，而configEpoch 每个master不一致，依次递增.
+ FAILOVER_AUTH_REQUEST和FAILOVER_AUTH_ack消息只有公共的消息头，没有消息体，也就是只有消息发送者节点的信息。
+ 1、只有分配了槽位的master可以投票。2、投票请求的currentEpoch必须大于master节点的currentEpoch，每个currentEpoch只会投一张选票。
+ slave切换master: 将flags设为master，移除原来master的slaves中的自己，将原来master的槽位加给自己，重新生成复制ID，将状态保存到node.conf，更新cluster.state为OK，将configEpoch更新为currentEpoch。
+ slave会丢弃epoch小于自身currentEpoch的ack，防止收到上一轮投票ack
+ 当slave收到大多数选票时，成功切换。如果NODE_TIMEOUT * 2内未收到大多数选票，NODE_TIMEOUT * 4后发起新一轮投票
+ 当slave发现满足主从切换的条件时，会延迟一小段时间后，再发起选举，延迟时间=500 milliseconds + random delay between 0 and 500 milliseconds + REPLICA_RANK * 1000 milliseconds，延迟时间的作用：1、由随机数组成，防止多个slave同时发起选举，导致票数相同。2、<font style="color:#DF2A3F;">REPLICA_RANK</font>和slave的复制偏移量有关，将slave根据复制偏移量从大到小排序，REPLICA_RANK从0开始递增，从而使复制偏移量大的slave延迟时间更短，先发起选举。
+ relica_rank不是强制的，如果为0的slave超时时间内没有当选，则下一个slave发起投票
+ slave当选为master后，递增configEpoch，将将集群节点信息发送给其它节点，其它节点收到后，发现configEpoch大于自己的，更新自己的集群节点信息
+ master没有拒绝投票的响应，忽略投票请求表示拒绝
+ master节点会持久化保存lastVoteEpoch,表示收到的选票中最大的currentEpoch，master会忽略小于自身lastVoteEpoch的选票，比如一个slave重新发起选举，递增currentEpoch,master如果投了新的投票，就不会同意老的投票请求
+ master节点会忽略configEpoch小于自身的投票请求，比如A是master, B,C是slave，A宕机（或网络分区），B当选master，B宕机，A重新上线，这时A的configEpoch小于集群大多数master的，因为B当选master时更新过一次槽位信息，所以只能C当选master。
+ 宕机重启，成为哪个master的slave：原来宕机的master节点重新上线后，发现它的所有槽位都分配给了其它master，于是它会成为它原来的最后一个槽位的master的slave，对于slave宕机重启也是一样
+ cluster-replica-no-failover：是否关闭自动故障切换，默认no，当master处于fail状态时，slave不会自动切换为master。仍可以通过cluster failover手动切换

### 副本迁移
+ 防止出现孤儿master，也就是slave都客观下线的master。没有客观下线的slave最多的master的slave会迁移给slave都客观下线的master
+ cluster-allow-replica-migration: 默认yes，开启副本迁移
+ 具体实现逻辑：集群中没有客观下线的slave最多的master的slave的其中node id最小的slave会成为acting slave，acting slave如果检测到了孤儿master，则它就会迁移过去（递增config epoch，修改节点信息，发送心跳）
+ cluster-migration-barrier：副本迁移时，源master至少保留多少个slave，默认1个

### 源码解析
#### 初始化集群
+ 根据cluster-config-file文件是否为空来判断是否是第一次启动集群，如果第一次启动，会生成node id
+ cluster nodes命令返回信息和节点配置文件内容相同，都是通过clusterGenNodesDescription
+ cluster nodes字段：第4个字段：为master时是"-"，slave是对应的master node id。第5个字段：act ping timestamp。6: last pong timestamp。7: configEpoch
+ 节点配置文件除了cluster nodes的内容，多了一行currentEpoch和lastVoteEpoch，这2个是集群级的字段，而configEpoch是节点级的字段，每个master节点对应一个，slave的configEpoch和master相同。
+ cluster-announce-human-nodename: 默认为空，可以配置为人类可读的nodename，会和nodeId加在一起使用
+ 执行--cluster create之前，cluster nodes只有当前节点，每台都是master
+ ping,pong,meet消息格式完全相同，只有类型不同，消息头中包含发送方的信息，比如node id，ip,port,slot,currentEpoch,configEpoch,repl offset；消息体中包含了gossip段，将发送方已知的集群节点的随机1/10会包含在gossip段中。为什么是1/10：为了在node_timeout*2时间内将pfail节点的信息同步给大多数节点，node_timeout/2时间内会交换2个ping包，因此node_timeout*2时间内交换8个包，总共8*node_num个包，8*node_num*(node_num/10)个节点entry，每个entry出现pfail节点的概率是1/node_num，8*node_num*(node_num/10)*(1/node_num)=node_num*80%，也就是80%的节点能收到pfail节点信息。
+ gossip段还包含了forget命令发送的节点信息，用于节点间同步待删除节点信息，会加到一个60s过期的黑名单中
+ cluster replicate：slave收到后，更新shardId、slaveof、flags等信息，通过client port与master建立连接，发起psync同步请求。<font style="color:#DF2A3F;">注意主从同步仍使用master的client port，而不是cluster port</font>。

**cluster meet**

+ 执行--cluster create时，redis-cli程序会使用第一台节点a 的ip port，向其它节点(比如其中一个b)发送cluster meet命令，b收到后，会创建a的clusterNode，其中的flags为handshake，nodeId暂时随机生成，并且添加到cluster.nodes，主线程轮询时，遍历cluster.nodes，主动和a建立连接，发送meet消息；
+ a收到meet消息后，也会创建一个flags为handshake，nodeId暂时随机生成的b的clusterNode，添加到cluster.nodes，并回复pong消息，将自己真实的nodeId发送给b，b收到pong后，将clusterNode的nodeId更新为真实ID
+ 因为a也添加了cluster.nodes，主线程轮询时也会主动建立一个和b的连接，发送ping,b收到后回复pong，将真实nodeId回复给a，更新。此时2个节点建立2个TCP连接，并且cluster.nodes中保存了对方节点信息。
+ a<-->b，a<-->c，a<-->d，acluster.nodes中保存了所有其它节点，因此会加到ping消息的gossip段中，最终所有其它节点两两服务发现，建立2个TCP连接

#### **redis-cli --cluster create**
+ main方法在redis-cli.c
+ --askpass，交互式输入密码
+ -x: 从标准输入读取数据，作为命令的最后一个参数
+ -X <tag>：从标准输入读取数据，作为命令中tag占位符的值
+ --cluster create创建集群，会根据节点数和cluster-replicas计算主节点的个数，nodeNum/cluster-replicas+1，至少3个，否则报错
+ 反亲和：--cluster create创建集群，会优先选择不同IP的节点为master节点，为同一master分配slave时，尽可能选择和master不同ip的slave节点，slave节点之间的ip也尽可能不同
+ <font style="color:#DF2A3F;">cluster addslots：</font>向master节点发送<font style="color:#DF2A3F;">cluster addslots</font> <font style="color:#DF2A3F;"><slotnum1> <slotnum2>...</font>，master节点收到后更新自身槽位信息，cluster addslots只能添加槽位，不能修改
+ <font style="color:#DF2A3F;">初始化configEpoch</font>：向每个节点发送cluster set-config-epoch <configEpoch>，给每个master节点初始化<font style="color:#DF2A3F;">不同的configEpoch，从1开始递增</font>，这时还没有服务发现，每台都是master。<font style="color:#DF2A3F;">cluster set-config-epoch只能在节点服务发现之前执行</font>，
+ <font style="color:#DF2A3F;">cluster meet</font>：获取第一个节点的ip端口号，向其它节点发送<font style="color:#DF2A3F;">cluster meet <firstNodeIp> <firstNodePort></font>，cluster meet的作用：必须管理员手动发送这条命令才能形成集群，防止因ip地址发生变化导致不同的集群中的节点形成集群。通过cluster meet，第一个节点知道其它所有节点，然后第一个节点发送gossip命令给其它节点，使其它节点之间相互知晓。redis集群每2个节点会建立2个TCP连接，比如a和b，会建立a->b，b->a，给a发送cluster meet b后会形成这2个连接。**<font style="color:#DF2A3F;">只能通过cluster meet或gossip来服务发现</font>**。
+ 等待1s，让节点间进行服务发现
+ <font style="color:#DF2A3F;">cluster REPLICATE：</font>向slave节点发送<font style="color:#DF2A3F;">cluster relicate <master node id></font>，slave收到后更新自身相应配置。
+ 等待cluster nodes一致：向每个节点发送cluster nodes，直到返回值相同

#### springboot lettuce
+ LettuceConnectionFactory: lettuce连接工厂，实现了springboot data redis中的RedisConnectionFactory接口，而springboot redistemplate根据该接口创建连接并使用，从而实现lettuce和springboot的整合
+ redistemplate.execute会执行LettuceConnectionFactory.getConnection()方法，获取连接，lettuce返回LettuceClusterConnection。得到连接后，可以在连接上执行redis命令方法，会获取调用xxCommands方法获取xxCommands对象，然后调用xxCommands对象的redis命令方法
+ 启动时，根据连接类型（比如集群类型）获取连接StatefulRedisClusterConnectionImpl；真正执行命令时，还会根据key计算槽位号，根据槽位号得到节点实例，然后根据节点实例获取对单台节点的连接StatefulRedisConnection。	
+ 初始化集群拓扑：先与配置中的每个节点建立连接StatefulRedisConnection，然后对每个节点执行cluster nodes，取该命令返回到节点集合和配置的节点集合的差集，对差集进一步建立连接，<font style="color:#DF2A3F;">因此可以只配集群中的部分实例，会根据cluster nodes进行服务发现</font>。
+ 刷新集群拓扑：如果开启了lettuce.cluster.refresh.adptive和lettuce.cluster.refresh.period，会创建定时任务刷新集群拓扑。和初始化的区别：初始化是在配置的节点基础上执行cluster nodes获取节点信息，然后取其中一个非fail节点最多的cluster nodes返回的节点信息；刷新是在<font style="color:#DF2A3F;">初始化或前一次刷新</font>的节点信息的基础上执行cluster nodes获取节点信息，然后取其中一个和老的节点信息nodeId变动最少的。
+ 连接池返回的StatefulRedisClusterConnectionImpl实际上对应了和集群拓扑中的每个节点的连接StatefulRedisConnection，<font style="color:#DF2A3F;">因此连接池不宜过大，真正连接数=连接池的个数*节点数</font>。
+ lettuce默认只向master发送命令，读命令不会发给slave。因为它根据cluster node返回的槽位信息更新Partitions.slotCache，而slave没有占用槽位

## SCAN、HSCAN、SSCAN、ZSCAN
+ DBSIZE: 获取key的个数
+ scan家族的命令和keys、hgetall、smembers的区别：scan每次只返回一部分值，对于大量key不会长时间阻塞redis主线程，而后者一次性返回所有的值，对于大量key会长时间阻塞redis主线程，导致其它客户端执行命令阻塞；scan因为每次只返回一部分值，数据一致性较差，数据集合可能在迭代的过程中改变，而后者原子执行，确保数据一致性。
+ scan和hcan, sscan,zscan的区别：scan是遍历当前数据库的所有key，后者遍历某个key对应的hash,set,zset集合，因此第一个参数多了key的名称，scan的参数没有key名称
+ 集群模式使用管道，使用上和单机模式一样，因为lettuce会在客户端计算key的槽位号，获取节点实例后，将管道中的命令全部发送给服务端后，再等待它们的返回结果。
+ scan传入游标，第一次传0，后面传上一次返回的游标值，当返回0时，遍历结束。
+ scan真正返回的个数不一定和count参数完全一致,甚至有可能返回0个元素（hint，提示值），但这并不代表遍历结束，只有游标为0时，遍历结束。count不指定时默认为10
+ 第一个返回值都是游标值，第2个返回值：scan返回key（没有val），hscan返回key val，sscan返回集合中的元素，zscan返回元素和分数
+ 当遍历大到需要使用hashtable存储的集合并且没有match时，实际返回的元素个数会大于count值，因为<font style="color:#DF2A3F;">scan以槽位为单位返回hashtable中的数据</font>，；当遍历小到足以使用intset，listpack存储的集合时，第一次就会返回所有元素；string没有存储优化
+ hash的entry键值对个数(hash-max-listpack-entries)超过512个，或最大entry(hash-max-listpack-value)大于64B时，会转成hashtable，否则listpack。set元素如果都是64位整数并且个数小于512个，使用intset，如果个数小于128个的非整数并且最大元素不超过64B，使用listpach，否则使用hashtable。zset如果个数小于128并且最大元素不超过64B，使用listpack，否则使用skiplist+hashtable
+ match省略时，默认为*。是先count查出来再使用match，因此如果match匹配到的很少，会导致多轮都返回0个元素.
+ match的pattern带上{}时，对于集群可以优化，会先根据大括号中的子串计算槽位，然后只查对应槽位所在的节点
+ scan可以通过type可以指定查哪些类型的key，类型通过type <key>获取，因此geohash和zset 的type都是zset。type和match一样，也是先count查出来再过滤，因此可能会导致多轮都返回0个元素
+ 游标保存了遍历的状态，服务端不保存任何遍历状态，因此可以多个客户端并发遍历；结束遍历时，客户端不需要告知服务端关闭游标
+ 如果遍历的过程中还在不断增加元素，可能会导致遍历无法结束
+ redis scan 命令底层原理（为什么会重复扫描？）：[https://blog.csdn.net/ldw201510803006/article/details/124052245](https://blog.csdn.net/ldw201510803006/article/details/124052245)

<font style="color:#DF2A3F;">游标变化</font>：

+ 初始值是0，后续从高位向低位递增，比如000,100,010,110,001,101,011,111。
+ 扩容后的遍历顺序是：0000,1000,0100,1100,0010,1010,0110,1110,0001,1001,0101,1101,0011,1011,0111,1111
+ 假设遍历到了110，发生扩容，并且rehash结束，110对应扩容后到0110，0110前面的都已经遍历，后面的没有遍历，所以正好不重复
+ 缩容后的遍历顺序是：00,10,01,11
+ 如果遍历到了110，发送缩容，并且rehash结束，下一个001取余后01，前面的全部遍历，后面的没有遍历，正好不重复
+ 如果遍历到了001，发送缩容，并且rehash结束，下一个101取余后01，而01对应扩容前到001和101，001槽位中的元素重复遍历
+ 如果2次scan之间正在扩容，取游标c在小表t0表对应槽位的数据，以及刚才的槽位在大表t1表对应的扩容槽位的数据，如果c对应的槽位已经扩容，那么可以在t1中查到，如果没有，可以在t0中查到。因为rehash是以槽位为单位迁移的，且rehash和scan都在主线程事件轮询中执行，同一时刻只会执行其中一个动作，所以数据要么在t0，要么在t1，不会重复和遗漏
+ 结论：<font style="color:#DF2A3F;">2次scan中间发生rehash，并且rehash未结束，不会重复读取和遗漏数据；2次scan中间发生扩容rehash，并且rehash结束，不会重复读取和遗漏数据；2次scan中间发生缩容rehash，并且rehash结束，可能会重复读取和遗漏数据，可以在客户端对key去重（也许是为什么没有slist的原因？）</font>；

```c
typedef struct {
    list *keys;   /* 遍历到的key,如果是hash或zset，包含了key和value,成对保存 */
    robj *o;      /* 如果是scan，则为NULL；如果是hscan,zscan,sscan,则为传入的key对应的value，hash/set/zset object */
    long long type; /* scan的type参数 */
    sds pattern;  /* match参数 */
    long sampled; /* 加进keys的key的个数，先统计sampled，再进行type,match过滤，所以可能会很多轮返回0个元素 */
} scanData;
unsigned long dictScanDefrag(dict *d, //要遍历的dict
                             unsigned long v, //客户端传的cursor
                             dictScanFunction *fn, //遍历到entry后的回调函数
                             dictDefragFunctions *defragfns,//为NULL
                             void *privdata)//scanData,保存了
{
    int htidx0, htidx1;
    const dictEntry *de, *next;
    unsigned long m0, m1;

    if (dictSize(d) == 0) return 0;

    /* This is needed in case the scan callback tries to do dictFind or alike. */
    dictPauseRehashing(d);

    if (!dictIsRehashing(d)) {
        //没有正在rehash，2次scan过程中可能没有rehash，也可能rehash但结束了，rehash分为扩容和缩容
        htidx0 = 0;
        m0 = DICTHT_SIZE_MASK(d->ht_size_exp[htidx0]);
        de = d->ht_table[htidx0][v & m0];
        while (de) {
            next = dictGetNext(de);
            fn(privdata, de);
            de = next;
        }
//游标变化：初始值是0，后续从高位向低位递增，
        //比如000,100,010,110,001,101,011,111。
    //扩容后的遍历顺序是：0000,1000,0100,1100,0010,1010,0110,1110,0001,1001,0101,1101,0011,1011,0111,1111
//假设遍历到了110，发生扩容，并且rehash结束，110对应扩容后到0110，0110前面的都已经遍历，后面的没有遍历，所以正好不重复
  //缩容后的遍历顺序是：00,10,01,11
//如果遍历到了110，发送缩容，并且rehash结束，下一个001取余后01，前面的全部遍历，后面的没有遍历，正好不重复
//如果遍历到了001，发送缩容，并且rehash结束，下一个101取余后01，而01对应扩容前到001和101，001槽位中的元素重复遍历
        
        v |= ~m0;
        v = rev(v);
        v++;
        v = rev(v);

    } else {
        htidx0 = 0;
        htidx1 = 1;
//2次scan之间正在扩容
        /* t0一定是小表，t1一定是大表，所以扩缩容情况一样 */
        if (DICTHT_SIZE(d->ht_size_exp[htidx0]) > DICTHT_SIZE(d->ht_size_exp[htidx1])) {
            htidx0 = 1;
            htidx1 = 0;
        }

        m0 = DICTHT_SIZE_MASK(d->ht_size_exp[htidx0]);
        m1 = DICTHT_SIZE_MASK(d->ht_size_exp[htidx1]);
        de = d->ht_table[htidx0][v & m0];
        //取游标在小表t0表对应槽位的数据
        while (de) {
            next = dictGetNext(de);
            fn(privdata, de);
            de = next;
        }
        /**
取刚才的槽位在大表t1表对应的扩容槽位的数据
        比如扩容前遍历顺序：00,10,01,11
        扩容后：000,100,010,110,001,101,011,111
        假如遍历到10时发生扩容，将10,010,110都读出来
        */
        do {

            de = d->ht_table[htidx1][v & m1];
            while (de) {
                next = dictGetNext(de);
                fn(privdata, de);
                de = next;
            }

            v |= ~m1;
            v = rev(v);
            v++;
            v = rev(v);

        } while (v & (m0 ^ m1));
    }

    dictResumeRehashing(d);

    return v;
}
```
